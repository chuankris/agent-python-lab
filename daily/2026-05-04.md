# 今日任务：写第一个 tool schema，约束关键词提取输入输出

日期：2026-05-04

## 今日目标
为一个最小工具能力写清楚输入输出契约，先完成第一个可读、可校验、可复用的 tool schema，推荐场景是 `extract_keywords`。

## 为什么做这个
你前面已经逐步碰了 FastAPI、Pydantic、日志、pytest 和 README。下一步进入 Agent 工程时，最容易乱的不是“调不调模型”，而是“工具到底收什么、回什么”。先把一个小工具的 schema 写清楚，后面接最小 tool-calling flow 时，接口层、模型层和测试层才有共同参照。这一步比直接堆调用代码更值。

## 任务说明
1. 先选一个足够小、不会分散注意力的工具能力，优先用 `extract_keywords`：输入一段文本，输出关键词列表。
2. 为这个工具补一份最小 schema。你可以用一种你当前最容易维护的形式来写，例如：
   - Python 里的 Pydantic 模型
   - 单独的 JSON Schema 文件
   - 放在 `README` 或模块注释里的结构化字段定义
3. schema 至少要写清楚这些内容：
   - 工具名
   - 工具用途的一句话说明
   - 输入字段有哪些
   - 每个输入字段的类型和是否必填
   - 输出结构长什么样
4. 输入尽量收敛，今天不要设计复杂参数。推荐最小输入只保留一个文本字段，例如 `text`；如果你想多练一点，也最多再加一个类似 `top_k` 的可选字段。
5. 输出也保持克制，优先返回固定结构，例如 `keywords: list[str]`，不要一上来设计很多元数据。
6. 如果你前几天的 FastAPI 或 `/task` 还没做完，今天也可以先只把 schema 单独放进一个清晰位置，不强求真正接入接口；重点是先把“工具契约”这件事落地。
7. 如果你已经有 `/task` 或类似入口，可以额外补 2 到 4 句注释或 README 说明，写清楚这个 schema 未来准备接到哪里，但今天不要继续把完整 tool-calling flow 一口气写完。

## 验收标准
1. 仓库里新增了 1 份明确可读的 tool schema，主题是 `extract_keywords` 或同等级别的最小工具。
2. schema 中清楚写出了输入字段、字段类型、必填约束和输出结构。
3. 设计足够小，整体工作能在 20 到 60 分钟内完成。
4. 如果你已经在用 Pydantic，schema 写法和现有项目风格基本一致；如果还没有现成代码，也能让人一眼看懂以后怎么接入。
5. 今天不追求真实模型调用，不追求复杂工具编排，只完成“工具契约”这一小步。
