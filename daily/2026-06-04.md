# 今日任务：把 `/task` 和第一个 tool schema 连成最小闭环

日期：2026-06-04

## 今日目标
让你当前的最小任务入口真正走通一次“输入文本 -> 判断是否调用工具 -> 返回结构化结果”的闭环，但只接一个工具、只覆盖一个最小场景。

## 为什么做这个
前两天你已经分别碰到了 `pytest` 和 tool schema，下一步最值得补上的不是继续堆新功能，而是把这些零散部件接起来。对 Agent 工程来说，真正容易失控的地方往往不是模型本身，而是“入口参数怎么进来、工具何时触发、结果如何回给调用方”这条链路不清楚。今天这一步很小，但它会直接训练你对最小 tool-calling flow、返回格式稳定性和接口边界的控制感。即使你前一天还没完全做完，也可以独立完成，因为今天只要求你围绕一个已有入口和一个工具定义做最小集成。

## 任务说明
1. 选定 1 个最小入口，优先使用当前的 `POST /task`；如果你的代码里还没正式挂路由，就围绕等价的处理函数完成同样的输入到输出闭环。
2. 只接 1 个工具，优先使用你昨天定义的 `extract_keywords` 或等价工具，不要新增第二个工具。
3. 先明确 1 条最小触发规则，例如“收到非空文本后固定调用一次工具”或“当请求里带某个 mode 字段时调用工具”，今天不要做复杂路由策略。
4. 把输入、工具参数、工具输出、接口响应这 4 层的字段名对齐一遍，避免 `text`、`query`、`content` 混用。
5. 工具执行逻辑只要做到最小可验证即可；如果你还不想写真正算法，可以先返回稳定的占位结构，但字段必须和 schema 对得上。
6. 补 1 条最小验证，优先是 `pytest`：断言请求能走到闭环，且响应里至少包含工具名、主要输出字段或调用结果状态中的任意 2 项。
7. 顺手把 README 或练习说明里的 1 处调用示例同步掉，让文档能说明这条最小链路现在是怎么工作的。
8. 今天不要扩展到多工具注册、外部模型接入、复杂调度、统一异常框架；只把 1 条最小 tool-calling flow 跑通。

## 验收标准
1. 已有 1 条从任务入口到单个工具再到响应输出的最小闭环。
2. 闭环中的字段命名在 schema、实现、测试或 README 中基本一致，没有明显各写各的。
3. 已完成 1 条最小验证，能证明这次请求确实触发了工具或返回了约定好的工具结果结构。
4. README 或对应练习说明已补充 1 处与当前闭环一致的输入输出示例或运行说明。
5. 整体工作量控制在 20 到 60 分钟，没有扩展成完整 Agent 框架。
