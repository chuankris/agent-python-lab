# 今日任务：整理最小 Agent 服务 README

日期：2026-06-09

## 今日目标
把你当前的最小 FastAPI/Agent 练习项目说明补到“新开一个终端也能照着跑”的程度，重点写清启动方式、`/task` 的最小请求示例、测试命令、格式化命令，以及当前这条最小 tool-calling flow 的边界。

## 为什么做这个
前几天你已经陆续碰到 `POST /task`、Pydantic 模型、日志、最小工具调用、`pytest` 和格式化工具。这个阶段如果 README 还是模糊的，后面每加一个小功能都会开始靠记忆维持，结果就是“代码能跑，但下周的自己很难接手”。  
今天这一步不引入新能力，而是把已经做过的能力收束成一份工程说明。它对 Agent 工程特别重要，因为 Agent 服务经常有输入格式、工具 schema、日志位置、失败场景这些“看代码才能知道”的隐性约定。把这些约定写出来，本质上是在练工程表达，而不是只练语法。

## 任务说明
1. 先以你仓库里现在真实存在的代码为准，确认最小可运行入口，不要为了写 README 反过来补一大堆新功能。
2. 在主 README 或当前练习目录 README 中，补一段“本地启动”说明，至少写清环境准备、安装依赖、启动命令。
3. 补一段最小接口使用示例，优先写 `POST /task`：请求体里有哪些字段、最小合法示例长什么样、返回结果至少会包含什么。
4. 如果当前 `/task` 已经接了最小 tool-calling flow，就用 2 到 4 句话说明这条链路现在做了什么；如果还没完全接上，就明确写“当前只做到哪一步”。
5. 补一段开发命令说明，至少覆盖 1 条测试命令和 1 条格式化或 lint 命令，让后续每天都能重复使用。
6. 顺手检查 README 里的字段命名是否和 Pydantic 模型、测试用例、接口返回保持一致；如果只差 1 到 2 处，小范围修正文案即可。
7. 今天不要扩展成“大改文档结构”“补完整架构图”或“重写项目介绍”，重点是把当前最小服务说明补清楚。

## 验收标准
1. 仓库中新增或更新了 1 处 README 说明，并且内容基于当前真实代码而不是预想实现。
2. README 中明确写出了本地启动命令。
3. README 中明确写出了 `/task` 的 1 个最小请求示例，且字段名与当前代码一致。
4. README 中明确写出了至少 1 条测试命令和 1 条格式化或 lint 命令。
5. README 中对当前最小 tool-calling flow 的能力边界有一句清楚说明，不会让人误以为它已经是完整 Agent。
6. 整体工作量控制在 20 到 60 分钟，没有演变成大规模重构。
