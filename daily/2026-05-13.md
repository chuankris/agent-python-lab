# 今日任务：实现最小 `GET /version`

日期：2026-05-13

## 今日目标
给当前最小 FastAPI 服务补一个 `GET /version` 接口，返回固定版本信息；如果昨天的 `GET /health` 还没完成，先用很短时间补到可访问，再继续今天任务。

## 为什么做这个
`GET /health` 解决的是“服务活着没有”，`GET /version` 解决的是“当前跑的是哪个版本、接口有没有稳定返回结构”。这是后面继续做 `/task`、Pydantic 响应模型、日志字段和最小测试时很常见的一类基础接口。它足够小，适合 20 到 60 分钟内完成；就算你昨天没做完，也能先补最小健康检查，再把今天的版本接口接上。

## 任务说明
1. 如果项目里还没有可启动的 FastAPI 入口，先补到服务能启动；如果 `GET /health` 还没落地，先补出最小健康检查，再继续今天的 `GET /version`。
2. 新增一个最小 `GET /version` 路由，只表达“当前服务版本”这一件事，不要提前把环境信息、构建时间、依赖列表或 Git 提交号一起塞进去。
3. 返回结构保持简单、字段命名统一，方便后面逐步过渡到 Pydantic 响应模型；今天可以先用固定值，但要让结构看起来像以后能稳定扩展。
4. 如果仓库里已经有测试入口，补 1 条围绕 `GET /version` 的最小 `pytest` 用例，只验证状态码和核心字段，不把今天扩展成完整测试日。
5. 如果 README 里还没有写清启动命令或接口验证方式，补最少几行，让你今晚打开仓库就知道怎么启动服务、怎么验证 `GET /version`。
6. 控制范围，不要顺手展开 `/task`、tool schema、日志重构或异常处理。今天只把“固定版本接口 + 可验证结果”做扎实。

## 验收标准
1. 本地已经有可访问的 `GET /version` 接口。
2. 访问该接口时，能返回一个简单、固定、可理解的版本结果。
3. 如果补了测试，`pytest` 至少有 1 条围绕 `GET /version` 的用例通过。
4. README 或相关说明里，至少写清 1 条启动命令，以及 1 种验证 `GET /version` 的方式。
5. 今天没有把任务膨胀成 `/task`、复杂日志、Pydantic 细化或工具调用实现，整体仍然是一轮短练习。
