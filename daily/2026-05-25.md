# 今日任务：让 `POST /task` 接上 1 个本地占位工具

日期：2026-05-25

## 今日目标
把现有的 `POST /task` 往前推进半步：只接 1 个你已经定义过或准备定义的最小工具，让请求进来后能够走到一次本地占位调用，并返回清晰、稳定的结构化结果。

## 为什么做这个
前几天你已经围绕 `POST /task`、tool schema、最小流程说明和轻量测试做了铺垫。接下来最自然的一步，不是立刻连真实大模型，而是先把“接口如何使用工具”这件事在本地跑通一次。对 Agent 工程来说，真正难的通常不是 API 能不能调，而是边界能不能先收清楚：输入是什么，选中了哪个工具，调用后返回什么。今天这一步做小一点，但价值很高。即使你前两天没全部完成，这个任务也依然能独立成立，因为你可以先用固定工具名或占位逻辑把闭环搭起来。

## 任务说明
1. 只选 1 个最小工具，不要扩成多工具注册表。建议继续用你前面已经写过或计划写的 `extract_keywords` 一类工具。
2. 在 `POST /task` 的当前流程里，补一个最小判断或直接固定调用：收到请求后，把文本交给这个本地占位工具处理。
3. 今天不要接真实模型，不要接外部 API。工具执行可以只是本地函数、假返回结果，或固定结构的占位实现。
4. 返回结果里至少要让人看清 3 件事：本次用了哪个工具、工具收到了什么核心输入、工具返回了什么结构化结果。
5. 如果你已经有 Pydantic 模型，优先把这次返回结构也补成一个最小响应模型；如果还没有，也至少保证字段命名统一、可读。
6. 补 1 条最小保护，二选一即可：
   - 为这个“`/task` -> 工具 -> 输出”闭环补 1 条 `pytest`；
   - 或补 1 条日志，让你在本地能看见本次请求命中了哪个工具。
7. 最后用 3 到 6 行更新 README 或相关说明，写清楚当前这个最小闭环还只是本地占位版本，不是真实模型调用。

## 验收标准
1. `POST /task` 已经不只是回显输入，而是能走到 1 个明确的本地工具占位调用。
2. 返回结果里能看出工具名、核心输入和结构化输出，而不是只有一段模糊文本。
3. 今天只实现单工具、单路径的最小闭环，没有扩展到真实模型接入或多工具编排。
4. 至少补了 1 条最小保护：1 条测试，或 1 条可观察的命中日志。
5. README 或相关说明里明确写出了：当前闭环是本地占位版，后续才会替换成真实 tool-calling。
