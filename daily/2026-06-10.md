# 今日任务：收紧 `/task` 最小响应结构

日期：2026-06-10

## 今日目标
给当前的 `POST /task` 补一份最小但稳定的响应结构定义：明确接口成功时至少返回哪些字段，并补 1 个对应的 `pytest` 用例，再把 README 里的示例和真实返回对齐。

## 为什么做这个
前面你已经碰过 FastAPI、Pydantic、最小 tool schema、`/task` 闭环、测试和 README。下一步不该急着加更复杂的 Agent 能力，而是先把“接口到底返回什么”收紧。  
对 Agent 工程来说，响应结构一旦模糊，后面的日志、测试、README、工具调用结果都会慢慢漂移。今天这个任务很小，但很关键：它会逼你把接口契约说清楚，把 Pydantic 模型、测试断言和文档示例对到一起。即使你前一天没完全整理完 README，这个任务也能独立推进，因为它只要求你围绕当前最小 `/task` 流程做一次收口。

## 任务说明
1. 先看当前 `POST /task` 真正返回了什么，不要先改实现，先确认现状。
2. 用 1 个最小响应模型或等价方式，把成功响应收紧到 2 到 4 个核心字段即可，例如任务输入、使用的工具名、工具结果、状态这类信息；字段名以你项目现状为准，不要硬套示例。
3. 如果现在返回里混着调试信息、临时字段或命名不一致，今天只收 1 次口，控制在最小改动，不做大改。
4. 补 1 个 `pytest` 用例，断言成功请求时返回结构里至少包含你定义的核心字段，而不是只断言状态码。
5. README 里如果已经有 `/task` 示例，请把示例响应同步到最新字段；如果还没有成功响应示例，就补 1 个最小示例。
6. 如果你发现 tool schema 名称、响应字段名、测试断言三者不一致，优先统一其中最关键的一组命名，不要顺手扩成重构。
7. 今天不要继续扩展多工具分发、异步调用、流式输出、数据库持久化或复杂异常体系；只把最小成功响应契约站稳。

## 验收标准
1. `POST /task` 的成功响应已有明确且稳定的最小结构。
2. 这个结构已通过 Pydantic 模型、类型定义或等价方式体现在代码里。
3. 至少新增或更新 1 个 `pytest` 用例，断言返回 JSON 的核心字段。
4. README 或练习说明中的 `/task` 成功响应示例与当前真实实现一致。
5. 工作量控制在 20 到 60 分钟，没有扩展成新的功能开发。
