# 今日任务：为现有接口补最小测试并整理 README

日期：2026-06-20

## 今日目标
给你当前的最小 FastAPI 练习补上 1 到 2 个 `pytest` 用例，优先覆盖已经完成的 `GET /health` 和 `GET /version`；如果你还没做完这两个接口，先补齐接口，再把测试和 README 收口。

## 为什么做这个
今天是周六，任务刻意做轻一点，但不能变成“只看不练”。你这阶段最需要建立的不是一次写很多功能，而是把一个很小的服务做成“能跑、能测、能说明”的闭环。

对 Agent 工程来说，这个习惯很重要。后面无论是接 `POST /task`、补 Pydantic 模型、加日志，还是接最小 tool-calling flow，都会依赖这种最小验证方式：接口行为先固定，再继续往上叠功能。对你从 Java/IoT 集成转过来也是一样，先把契约和验证站稳，比一开始就追求复杂流程更划算。

## 任务说明
1. 继续在你已经使用的练习目录里做，不要新开一套项目。优先使用 `exercises/day-001-project-skeleton/`。
2. 先检查本地是否已经有 `GET /health` 和 `GET /version`。如果其中一个还没完成，先补齐到“本地能访问”的程度，但今天不要继续扩到 `POST /task`。
3. 使用 `pytest` 为现有接口补最小测试。最低要求是 1 个测试；如果时间够，推荐把 `/health` 和 `/version` 都覆盖掉。
4. 测试重点放在最基本的接口契约上：状态码、返回 JSON 结构、关键字段值。不要为了“写得完整”引入复杂夹具或重构很多目录。
5. 如果 `/version` 已经用了 Pydantic 响应模型，测试里顺手确认字段名和返回结构与模型一致；如果还没用模型，今天不要硬加很多字段，保持最小即可。
6. 更新练习目录下的 `README.md`，补上当前已支持的接口列表、最简单的启动方式，以及一条测试运行命令。
7. 如果还有 10 分钟左右余量，再做一个很小的整理动作即可：例如统一 README 里的命令格式，或者把测试文件命名改得更清楚。不要顺手开启日志、tool schema、格式化器接入等新主题。

## 验收标准
1. 本地至少有 1 个 `pytest` 用例可以运行通过。
2. 已完成的接口至少有 1 条被测试覆盖，推荐覆盖 `/health` 和 `/version`。
3. `README.md` 已写清楚如何启动服务、如何运行测试、当前有哪些已完成接口。
4. 今天的改动总量控制在 20 到 60 分钟，没有把任务扩展到 `/task`、日志系统或 tool-calling。
