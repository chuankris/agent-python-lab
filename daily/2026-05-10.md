# 今日任务：周末巩固最小 FastAPI 骨架和说明

日期：2026-05-10

## 今日目标
用一次轻量整理，把仓库对齐到“可以开始最小 Agent 服务开发”的状态：确认或补齐最基础的 FastAPI 目录骨架，并把 README 里的启动、自检和下一步范围写清楚。

## 为什么做这个
这两周的主线已经走到 FastAPI、Pydantic、日志、pytest、tool schema、格式化和 README。周末不一定要继续加新接口，更划算的是做一次收口：把目录、说明和最小运行约定先对齐。这样你下周继续做 `/task`、模型校验或最小 tool-calling flow 时，不会先卡在“项目该从哪里起步”“命令该怎么跑”这种低价值问题上。即使你前几天的代码任务还没补完，今天这件事也照样成立，因为它本质上是在给后续练习铺稳定地基。

## 任务说明
1. 先检查仓库当前是否已经有最小服务骨架；如果还没有，就只补最基础目录和占位文件，例如 `app/`、`tests/`，不要一口气展开完整业务代码。
2. 如果你已经写过 FastAPI 入口，今天只做整理，不追加复杂功能；如果还没写，可以只放一个最小入口文件占位，并在 README 里标明“下一步准备补 `GET /health` 或 `/task`”。
3. README 至少补清 3 件事：这个仓库当前处于什么阶段、下一个编码目标是什么、改完代码后先跑哪条自检命令。如果你昨天已经写过一部分，就以真实状态为准继续收紧。
4. 如果项目里已经接了 `ruff` 或准备接 `pytest`，把命令写成固定小节；如果还没落地，也可以先明确约定计划使用的命令，但不要为了今天这个任务反向扩展太多工具链。
5. 给未来的 `/task` 和 tool schema 留出清晰位置，例如在 README 或目录里约定 `app/schemas/`、`app/tools/` 会放什么；重点是命名和边界清楚，不是今天就写完实现。
6. 把工作量控制在“整理骨架 + 更新说明 + 最小占位”这个级别，不补真实业务逻辑，不写大段解决方案代码。

## 验收标准
1. 仓库已经有最小可理解的服务目录骨架，或者 README 明确写清当前还缺哪些骨架。
2. README 已说明当前阶段、下一步编码目标，以及至少 1 条启动或自检命令。
3. 对 `app/`、`tests/`、`app/schemas/`、`app/tools/` 这类目录的用途有最小约定，后续继续补代码时不会混乱。
4. 今天没有扩展成大功能开发，整体能在 20 到 60 分钟内完成。
