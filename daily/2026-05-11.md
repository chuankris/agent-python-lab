# 今日任务：初始化最小 FastAPI 服务骨架

日期：2026-05-11

## 今日目标
在仓库里补出一个最小可理解的 FastAPI 服务骨架，为后面的 `GET /health`、`/task`、Pydantic 模型、日志和 pytest 留出清晰位置。

## 为什么做这个
前几天的任务已经围着 FastAPI、Pydantic、日志、pytest、tool schema、格式化和 README 打转，但从仓库现状看，真正的服务代码骨架还没有落地。今天先把目录、入口和最小运行约定立住，比继续空谈接口设计更值钱。即使你前几天有漏做，今天也能直接开始，因为这一步本身就是后续所有练习的共同起点。

## 任务说明
1. 在仓库中创建最小服务目录骨架，至少让人一眼能看懂后面准备把代码放在哪里。建议优先考虑这些位置：`app/`、`app/main.py`、`app/schemas/`、`app/tools/`、`tests/`。
2. `app/main.py` 里只做“服务入口”这一件事：放一个最小 FastAPI 应用实例，并让后面的路由可以自然往这里继续加。今天不要顺手把 `/task`、tool-calling flow 或复杂依赖一起写进去。
3. 如果你愿意补一个占位接口，最多只放一个最小 `GET /health`，目标是验证服务能启动，不要扩展成多接口开发。
4. 给 `tests/` 留出最小测试位置即可；如果时间足够，可以只补最基础的测试文件骨架或一条最小启动验证，但不要把今天做成完整测试日。
5. 检查 `README.md` 是否需要补 3 到 6 行说明，写清楚当前仓库已经进入“最小 FastAPI 服务骨架已落地/待继续补接口”的阶段，以及最基本的启动方式。说明要和真实状态一致，不要假装已经有完整 Agent 流程。
6. 如果仓库里还没有 Python 依赖说明，可以只补到“后续准备使用 FastAPI、uvicorn、pytest、ruff”这个粒度，不要求今天把所有工具链一次性配齐。
7. 把工作量控制在 20 到 60 分钟。今天的重点是把后续编码的落点定清楚，不是把功能一次做完。

## 验收标准
1. 仓库里已经出现最小可理解的 FastAPI 服务骨架，至少包含入口文件和后续代码目录位置。
2. `app/main.py` 已经能代表服务入口；如果补了 `GET /health`，本地访问时应能返回一个简单成功状态。
3. `tests/` 已经有明确落点，后面可以继续补 pytest，而不是临时再想目录怎么放。
4. `README.md` 的说明与仓库真实状态一致，能看出当前是在搭最小服务骨架，而不是已经完成完整 Agent 服务。
5. 今天没有扩展成复杂业务开发，整体能在一次短练习里完成。
