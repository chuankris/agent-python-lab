# 今日任务：为最小 FastAPI 练习接入格式化，并收口 README

日期：2026-05-28

## 今日目标
给你当前正在推进的最小 FastAPI / Agent 练习补 1 个基础格式化工具，并把 README 里最容易漂掉的运行说明收口成几行可以直接照做的文字。

## 为什么做这个
这几天你已经在围绕 `POST /task`、Pydantic 校验、最小 tool 流程和 `pytest` 打基础。下一步最自然的动作，不是继续往下堆功能，而是先把工程习惯立住。对 Agent 工程来说，很多小项目后面变乱，并不是因为功能太难，而是因为格式、命名、运行方式没有尽早固定。今天这一步很小，但收益很稳定：以后你每次改接口、模型或测试时，都能更快发现“只是风格乱了”还是“真的改坏了”。即使你前一天没补完测试，今天也照样能做，因为格式化和 README 收口本身就是独立任务。

## 任务说明
1. 只选 1 个最小工具接入，优先选 `ruff format`、`black` 或你已经准备使用的格式化方案，不要同一天同时折腾太多工具。
2. 如果你已经有练习代码，就直接对当前最小 FastAPI / tool / test 相关文件应用这个工具；如果你还没有完整代码，也可以先把工具配置到准备练习的目录里。
3. 今天重点是“先跑通一次格式化命令”，不是追求完整 lint 体系。能成功执行 1 条格式化命令就够，不必同时补全所有检查规则。
4. 顺手看一眼你最近在动的文件，把 1 到 2 处最明显的可读性问题一起收掉，例如导入顺序、过长行、命名不一致，或 README 里的命令写法不统一。
5. 在 README 或练习目录说明里补 3 到 6 行，写清楚两件事：现在用什么命令做格式化；改完 `/task`、schema 或测试后，建议先跑哪条最小验证命令。
6. 如果你还没有 `README` 的细分练习说明，也可以把说明写到当前练习目录自己的 `README.md`，不要为了这件事大改仓库结构。
7. 今天不要扩展新接口，不要新增复杂功能；目标只是把“能继续迭代”的工程地基再压实一点。

## 验收标准
1. 已为当前最小练习接入并实际运行了 1 个格式化工具。
2. 至少有 1 处代码或说明文档因为这次整理变得更统一、更易读。
3. README 或练习说明里已经写明格式化命令，以及 1 条最小验证命令。
4. 今天没有把任务扩成完整 CI、复杂 lint 规则或大规模重构。
5. 你能用一句话说明：以后改 `/task`、Pydantic 模型或 `pytest` 前后，应该先跑哪条格式化命令。

- 你当前正在推进的 `app/`、`tests/` 或练习目录
