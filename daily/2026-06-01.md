# 今日任务：为最小 `/task` 流程补空输入校验和清晰错误返回

日期：2026-06-01

## 今日目标
围绕你当前最小的 `POST /task` 链路，补 1 个空输入或非法输入场景的处理，并让接口返回清晰、稳定、可验证的错误信息。

## 为什么做这个
前面你已经碰过 FastAPI、Pydantic、logging、pytest、tool schema 和最小 tool flow。下一步不该急着加更复杂的能力，而是先把“坏输入来了会怎样”这件事补齐。真实 Agent 服务里，最常见的问题不是 happy path 跑不通，而是空文本、缺字段、格式不对时返回含糊，最后让测试、日志和调用方都难受。今天这一步很小，但它会直接提升你对接口边界、Pydantic 校验和错误表达的掌控感。即使你前几天有任务没做完，今天也能独立完成，因为你只需要围绕一个最小入口处理 1 类错误。

## 任务说明
1. 先选定 1 个最小入口，优先是 `POST /task`；如果你现在还没正式写到这个接口，就选一个等价的请求模型校验入口。
2. 只处理 1 类最常见的坏输入，三选一即可：请求里缺少核心字段、文本是空字符串、文本只有空白字符。
3. 明确你希望返回什么错误表现，至少想清楚 3 个点：状态码、错误信息里要不要点名字段、返回结构是否和当前项目风格一致。
4. 如果你已经在用 Pydantic，优先让模型承担基础校验；如果还没做到这一步，也可以先在路由层做最小判断，但不要把今天扩成完整重构。
5. 补 1 条最小验证，优先用 `pytest` 写一个坏输入场景；如果你还没把测试环境跑顺，也可以先手动发一次请求并把预期错误写进 README 或练习说明。
6. 顺手检查 1 处相关输出是否同步，例如错误日志字段、README 示例、测试断言里的字段名，不要让“代码里一个名字、文档里另一个名字”继续漂着。
7. 今天不要新增第二个错误分支，不要设计统一异常框架，不要接更复杂的 tool 调度；只把 1 个坏输入场景收口清楚。

## 验收标准
1. 已明确并实现 1 个空输入或非法输入场景的处理。
2. 错误返回至少说明了“哪里错了”或“哪个字段有问题”。
3. 已完成 1 条最小验证：1 个 `pytest` 场景，或 1 次手动请求记录。
4. 相关的 1 处测试、日志或 README 示例已同步，没有明显自相矛盾。
5. 今天的改动仍然是一个 20 到 60 分钟的小任务，没有扩成完整异常体系设计。
