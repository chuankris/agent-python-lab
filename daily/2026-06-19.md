# 今日任务：实现版本接口并收敛返回模型

日期：2026-06-19

## 今日目标
在你的最小 FastAPI 练习里实现 `GET /version`，并用一个最小 Pydantic 响应模型把返回结构固定下来。

## 为什么做这个
从 `GET /health` 往后走，最自然的一步不是立刻把事情做复杂，而是再补一个更像真实服务的稳定接口。`/version` 很小，但它能把路由、固定响应、模型约束和 README 说明串起来。

这也是你从 Java 服务思维切到 Python Agent 工程时很值得建立的习惯：先把接口契约收紧，再继续往 `/task`、tool schema 和最小 tool-calling flow 推进。后面只要开始接请求体、响应体和工具结果，这种“先定结构”的动作会一直复用。

如果你前一天的 `/health` 还没做，也没关系。今天先花几分钟把 `/health` 补齐，然后继续做 `/version`，本任务仍然成立。

## 任务说明
1. 进入你当前的最小练习目录，优先继续在 `exercises/day-001-project-skeleton/` 上迭代，不要重新开第二套目录。
2. 在 FastAPI 入口里增加 `GET /version` 路由，先返回一个固定版本号；版本值可以直接写死，例如 `0.1.0`，今天不要引入环境变量或打包元数据读取。
3. 为这条接口补一个最小响应模型，建议只保留 1 到 2 个字段，例如 `version`，或者 `version` 加 `app_name`；重点是让你开始用 Pydantic 约束输出结构，而不是堆字段。
4. 如果你之前把模型都写在 `main.py`，今天可以只做一次很小的整理：把响应模型抽到 `schemas.py`，但不要顺手做大规模重构。
5. 补 1 个最小 `pytest` 用例，验证 `GET /version` 返回 `200`，并且字段名和字段值符合你的约定。
6. 更新练习目录里的 README，补上这条接口的访问示例，并用一句话说明为什么现在要开始引入响应模型。
7. 今天不要继续扩成 `POST /task`、不要接日志、不要写 tool schema；重点是完成“稳定接口 + 最小模型 + 最小测试”这个闭环。

## 验收标准
1. `GET /version` 已可访问，并返回固定 JSON 结构。
2. 这条接口已经使用最小 Pydantic 响应模型，而不是裸字典随意返回。
3. 至少有 1 个 `pytest` 用例覆盖 `/version` 的成功场景。
4. README 已补充 `/version` 的访问示例和一句简短说明。
5. 整体工作量控制在 20 到 60 分钟，没有顺手扩写 `/task`、日志体系或 tool-calling 逻辑。
