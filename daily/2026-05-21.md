# 今日任务：为 `POST /task` 补齐成功/失败配对测试

日期：2026-05-21

## 今日目标
围绕当前的 `POST /task`，补一组最小但成对的 `pytest` 用例，至少固定 1 条成功路径和 1 条失败路径，让接口行为开始有可回归的保护。

## 为什么做这个
前一天如果你已经给 `/task` 加了输入校验和标准错误响应，今天最自然的下一步不是继续改接口逻辑，而是先把这些行为写成测试。Agent 服务后面会继续接日志、tool schema 和最小 tool-calling，如果现在没有测试护栏，后面每加一层都容易把已有行为改坏。就算你昨天没做完，今天这件事也依然成立：先挑一个已经稳定的成功输入和一个明确的失败输入，把它们固化下来，收益很直接。

## 任务说明
1. 先确认当前 `/task` 已经至少能稳定处理 1 个成功请求；如果还不稳定，先收敛到一个最小可运行版本，再开始写测试。
2. 新增 1 条成功路径测试，重点检查状态码、返回 JSON 的核心字段，以及这些字段的类型或关键值是否符合你的当前约定。
3. 新增 1 条失败路径测试，建议选“空字符串”“缺少必填字段”或“字段类型错误”里最容易复现的一种，不要一口气铺太多分支。
4. 如果你已经有统一错误响应结构，测试里顺手断言 `code`、`message` 这类关键字段存在；如果还没有完全统一，就先只固定你今天已经实现稳定的部分。
5. 把测试命名写得能看出意图，例如一眼能分辨“正常创建任务”和“非法输入被拒绝”，避免未来回看时还要重新理解。
6. 跑一遍这 2 条测试；如果仓库里暂时还没完整测试命令，也至少把本次使用的执行方式补 2 到 3 行到 README 或测试文件注释附近。
7. 今天不要扩展到真实模型调用，不要新增更多 API，不要开始写 tool schema；先把 `/task` 当前行为用测试钉住。

## 验收标准
1. `tests/` 下至少新增 2 条和 `POST /task` 直接相关的 `pytest` 用例。
2. 至少覆盖 1 个成功场景和 1 个失败场景，而不是只测单边。
3. 成功用例会校验核心响应字段；失败用例会校验状态码，并尽量校验错误响应结构。
4. 你能用一句话解释每个测试分别在防什么回归。
5. 今天的工作量仍控制在 20 到 60 分钟内，没有扩展到 tool-calling、复杂日志体系或额外重构。
