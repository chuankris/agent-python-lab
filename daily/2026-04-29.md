# Day 004 / 2026-04-29

## 今日目标
为当前 FastAPI 小项目增加一个最小可用的 `POST /task` 接口，先接收一段文本输入，再返回一个结构化结果，为后面的 Pydantic、日志、测试和 tool schema 打基础。

## 为什么做这个
前两天你已经在熟悉“服务能启动”和“简单路由能访问”。今天往前走半步，把接口从只读的 `GET` 推进到带请求体的 `POST`，这是后面做 Agent 输入处理的最小起点。它还不涉及真实模型调用，但会先把“输入进来 -> 服务处理 -> 结构化输出”这条链路搭起来。

## 任务说明
1. 在现有 FastAPI 应用里新增一个 `POST /task` 路由。
2. 这个接口先只处理一个很简单的场景：接收一段任务文本，并返回一个结构化 JSON 结果。
3. 返回结果里至少保留“原始输入内容”和一个你自己定义的简单状态字段，让响应看起来像后续 Agent 流程的雏形。
4. 如果你昨天的 `/version` 还没做，今天可以先确保项目里至少有一个可启动的 FastAPI 入口，再继续做 `POST /task`；但今天的主要验收对象仍然是 `POST /task`。
5. 如果你已经知道怎么用 Pydantic，可以顺手用最小请求模型和响应模型来组织这个接口；如果还没完全熟悉，也可以先做最小可运行版本，明天再专门收口模型校验。
6. 顺手在 `README.md` 里补一小段说明：如何启动服务，以及如何用 `curl`、Swagger UI 或其他方式验证 `POST /task`。

## 验收标准
1. 本地可以启动 FastAPI 服务。
2. `POST /task` 能接收一段文本输入。
3. 接口能返回结构化 JSON，而不是只返回一段裸字符串。
4. `README.md` 至少写清楚一种验证 `POST /task` 的方式。
5. 整体改动控制在 20 到 60 分钟内可完成。
