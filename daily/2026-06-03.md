# 今日任务：定义一个最小可用的 tool schema

日期：2026-06-03

## 今日目标
为你当前的最小 Agent 练习补 1 个清晰、可验证的 tool schema，优先用“提取关键词”这类简单能力作为例子，但今天只做 schema 和输入输出约定，不写完整工具执行逻辑。

## 为什么做这个
你前面已经碰到 FastAPI、Pydantic、日志和最小测试，下一步最值得补的是“工具长什么样、参数怎么收、结果怎么表达”。真实 Agent 工程里，tool 调用经常不是先卡在模型，而是先卡在 schema 含糊：字段名不稳、必填项不清楚、描述写得太虚，最后接口、测试和 README 都对不上。今天把 schema 单独立住，能帮你先练清楚数据边界和命名，再为后面的最小 tool-calling flow 打底。即使你前几天没全部完成，这个任务也能独立做，因为它更像是在给后续能力立“合同”。

## 任务说明
1. 先选 1 个最小工具能力，优先推荐 `extract_keywords`，输入保持简单，例如一段文本和可选数量限制。
2. 只定义 1 份 schema，可以是 Pydantic 模型、Python 字典描述，或你项目里约定的 schema 结构；关键是字段要明确，不要同时尝试多种表示法。
3. schema 至少说清楚 4 件事：工具名、用途说明、输入字段、哪些字段必填。
4. 如果你已经在用 Pydantic，优先把字段类型、默认值、简单约束写清楚；如果还没整理到模型层，也可以先在单独文件里写出最小结构定义。
5. 补 1 个最小样例，说明一次合法输入长什么样，以及你期望工具返回的结果结构大概长什么样；样例可以放在 README、注释或练习说明里。
6. 顺手统一 1 处命名，例如 `keywords`、`items`、`text`、`query` 这些词只选一套，不要 schema、测试、README 各写各的。
7. 今天不要把 `/task` 真正接到 tool 上，不要引入外部模型，不要扩成“多工具注册系统”；只把 1 个工具 schema 定义清楚。

## 验收标准
1. 已新增 1 个最小 tool schema，能看出工具名、用途和输入字段定义。
2. schema 中至少有 1 个必填字段，字段类型或约束是明确的。
3. 已补 1 个合法输入样例，以及 1 个预期输出结构说明。
4. README、练习说明或代码注释中有 1 处内容已与 schema 命名保持一致。
5. 整体工作量控制在 20 到 60 分钟，没有扩展成完整 tool 调度实现。
