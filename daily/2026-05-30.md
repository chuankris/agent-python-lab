# 今日任务：收口最小 tool flow 的字段命名

日期：2026-05-30

## 今日目标
把你当前最小 `POST /task -> tool -> 输出` 这条链路里最容易漂掉的 2 到 3 个字段名收口一致，并补 1 条最小验证。

## 为什么做这个
这几天你已经碰过 `POST /task`、Pydantic、日志、`pytest`、tool schema、格式化和 README。这个阶段继续堆功能，收益不一定高；更值得做的是把已经出现的最小闭环再压实一点。很多 Agent 服务后面变难维护，不是因为能力太复杂，而是因为请求体、tool schema、返回结果、日志里的字段名各写各的。今天这一步偏轻量，但很像真实工程里的“周末收口”：不扩新功能，只把命名和验证固定住。即使你前一天没完全整理 README，今天也能独立完成。

## 任务说明
1. 选 1 条你已经能跑通或接近能跑通的最小链路，例如 `POST /task` 调 1 个本地占位 tool，或只经过 schema 校验再返回结果。
2. 找出这条链路里最关键的 2 到 3 个字段名，例如 `text`、`keywords`、`tool_name`、`result`、`request_id`，只处理最影响理解的那几个，不要一次性大改所有命名。
3. 对照看 4 个位置是否一致：Pydantic 模型、tool schema、接口返回、日志或 README 示例。如果同一含义出现了两个名字，今天就统一成一个。
4. 如果你暂时还没有完整日志，也可以把第 4 个位置换成测试断言或接口示例；重点是让“输入字段名”和“输出字段名”不要各说各话。
5. 补 1 条最小验证，二选一即可：写 1 个 `pytest` 断言字段名符合预期，或手动发 1 次请求并记录实际返回字段。
6. 顺手清掉 1 处最明显的小问题，例如 README 示例字段名过期、schema 描述和代码不一致、日志里字段拼写不统一。
7. 今天不要新增新接口、不要接入更复杂的 tool、不要把任务扩成大重构；目标只是把最小 tool flow 的命名收口清楚。

## 验收标准
1. 你已经明确并统一了最小链路里 2 到 3 个核心字段名。
2. 至少有 2 个位置保持一致，例如模型和返回、schema 和测试、返回和 README。
3. 已完成 1 条最小验证：1 个 `pytest`，或 1 次手动请求记录。
4. 至少修掉了 1 处“同一含义多个名字”的小问题。
5. 今天没有扩展新功能，而是把已有最小 Agent 流程整理得更稳定。
