# 今日任务：为 `/task` 补一组最小测试用例

日期：2026-05-07

## 今日目标
围绕当前的 `/task` 或等价入口，补上 2 到 3 个最小测试用例，覆盖至少 1 个正常场景和 1 个异常场景，让这个最小 Agent 服务开始具备“改动后能快速回归验证”的基础。

## 为什么做这个
这几天你已经逐步碰到了 FastAPI、Pydantic、日志、tool schema 和最小 tool-calling flow。下一步不该急着继续加功能，而是先把“现有行为能不能稳定验证”补起来。对 Agent 工程来说，测试不是锦上添花，它直接决定你后面敢不敢继续重构接口、调整 schema、补日志和改 README。即使你前两天的实现还不完整，今天也可以先围绕已经跑通的那一小段流程把测试立住，不会断档。

## 任务说明
1. 先盘点你当前 `/task` 的最小可验证行为，只选 2 到 3 个最值得测的点，不要一口气铺满。推荐优先级是：正常请求返回结构、空输入或缺字段报错、工具执行结果被正确放进响应。
2. 如果你已经有 FastAPI 的测试客户端，就直接沿用；如果还没有，就只补最小测试骨架，不要顺手扩展成完整测试框架。
3. 至少写 1 个正常场景测试。重点不是断言很多细节，而是确认这条链路真的走通了：请求进来、校验通过、工具流被触发或伪触发、响应结构符合预期。
4. 至少写 1 个异常场景测试。优先选择昨天补过的边界，例如空字符串、缺少 `text` 字段、字段类型不对，或者工具层抛错后接口返回清晰错误。
5. 如果你的工具调用目前还是本地规则或假实现，测试里也保持这个范围，不要为了“更真实”去接 LLM SDK、联网服务或复杂 mock 体系。
6. 时间够的话，再补第 3 个测试，用来覆盖一个你最担心后续回归时被改坏的点，例如日志相关返回字段、默认参数行为，或错误响应格式。
7. 顺手检查一下测试文件命名、断言可读性和输出格式，让别人一眼能看懂你在保什么行为；今天不追求参数化、fixtures 抽象或大规模重构。
8. 如果 README 里还没写怎么跑测试，可以补 2 到 4 行最小说明；但今天主任务仍然是把测试补起来，不要把重心切到文档大修。

## 验收标准
1. 已新增或补全 2 到 3 个 `pytest` 用例，且至少包含 1 个正常场景和 1 个异常场景。
2. 测试覆盖了当前 `/task` 或等价入口的真实行为，而不是只测独立辅助函数。
3. 至少有 1 个断言能验证响应结构中包含工具调用后的结果或清晰错误信息。
4. 测试范围控制在当前最小服务内，没有引入新的外部依赖、真实模型调用或过度复杂的 mock。
5. 整体工作量控制在 20 到 60 分钟，完成后你能用一条命令重新跑这组测试。
