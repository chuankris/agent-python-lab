# 今日任务：给 `POST /task` 补最小 Pydantic 模型和 1 条失败用例

日期：2026-05-16

## 今日目标
把当前最小 `POST /task` 接口再往前推进一小步：补一个请求模型、补一个响应模型，并用 1 条最小测试验证“无效输入会被拦住”。

## 为什么做这个
前一天如果你已经把 `POST /task` 跑通，下一步最自然的动作就是把“随便收一个文本”升级成“有明确输入结构和输出结构”的接口。这一步会直接连接到后面的日志、异常处理、tool schema 和最小 tool-calling flow；如果你昨天没做完，也依然能先补一个最小 `/task` 再继续今天的模型校验，不会断档。周末把任务收窄到 Pydantic 模型、1 条失败用例和少量 README 说明，负担更轻，但含金量很高。

## 任务说明
1. 如果仓库里还没有可启动的 FastAPI 服务，先补到“服务能启动，且有一个最小 `POST /task` 路由”这个程度，再继续今天的任务。
2. 为 `POST /task` 增加一个最小请求模型，只保留当前真正需要的字段，不要提前把优先级、标签、调用参数之类的扩展字段一口气塞进去。
3. 为 `POST /task` 增加一个最小响应模型，让返回结果的字段命名更稳定，方便后面继续接日志、状态字段和工具调用结果。
4. 明确 1 个最小无效输入场景，例如空字符串、缺少核心字段，或字段类型不对；今天只挑 1 种场景做透，不要把任务扩展成完整校验矩阵。
5. 补 1 条最小 `pytest` 用例，验证接口在这个无效输入场景下会返回清晰的失败结果；如果你还没配好测试环境，先把测试文件和断言思路搭出来。
6. README 里补几行说明：这个接口当前期望什么输入、返回什么核心字段，以及你用什么命令或请求方式验证了无效输入场景。
7. 今天不要引入真实大模型调用、不要实现 tool schema、不要铺开结构化日志；把“Pydantic 基本校验 + 失败输入测试”这一小段链路做扎实就够了。

## 验收标准
1. 本地已有可访问的 `POST /task` 接口，且它的请求体/响应体已经开始使用 Pydantic 模型描述。
2. 至少有 1 个无效输入场景会被接口明确拦住，而不是静默吞掉或返回含糊结果。
3. 至少有 1 条围绕这个失败场景的最小 `pytest` 用例；如果环境已可运行，测试应能通过。
4. README 或相关说明里，已经写清这个接口当前的最小输入、最小输出，以及 1 种手动验证方式。
5. 今天的改动仍然控制在 20 到 60 分钟内，没有扩展到真实 tool-calling、复杂日志或完整异常体系。
