# 今日任务：为最小 `/task` 流程补 1 条 pytest 验证
日期：2026-06-02

## 今日目标
围绕你当前最小的任务入口，补 1 条真正会执行的 `pytest` 用例，优先覆盖一次“输入进入 -> 触发最小处理 -> 返回结构化结果”的完整链路。

## 为什么做这个
前面几天如果已经做了 FastAPI、Pydantic、日志和最小 tool schema，下一步最值当的不是继续堆功能，而是先把一条可重复验证的测试链路立住。对 Agent 工程来说，很多问题不是代码完全写不出来，而是改完以后不知道有没有把原来的输入输出格式弄坏。今天只补 1 条测试，目标很小，但它会直接训练你对接口行为、返回结构、测试边界和回归验证的感觉。即使你前面的任务没全做完，这一步也仍然能独立完成，因为你只需要围绕当前仓库里已经存在的那个最小入口来验证。

## 任务说明
1. 先选定 1 个最小入口，优先是 `POST /task`；如果你还没把接口搭起来，就选一个等价的最小函数，比如“接收文本后返回结构化结果”的处理函数。
2. 只补 1 条 `pytest` 用例，不求覆盖全部情况。优先顺序如下：正常输入返回结构化结果；无 tool 时仍能返回稳定格式；已有最小 tool schema 时能走通一次调用。
3. 测试里至少断言 3 件事：状态是否成功或函数是否正常返回、返回 JSON 或字典里是否有稳定字段、字段命名是否和你当前 README/注释里的说法一致。
4. 如果当前代码里已经有 Pydantic 模型，优先断言模型落地后的字段；如果还没有，就先断言最小返回结构，不要把今天扩展成重构任务。
5. 如果当前有日志输出，顺手检查 1 个最关键的日志点是否还保留，例如收到请求、进入处理流或返回结果；没有的话今天可以不额外加日志系统。
6. 把 README 或练习说明里与这个入口相关的 1 处示例同步一下，避免“测试断言一种格式，文档写另一种格式”。
7. 今天不要追求补第二条测试、不要追求把测试基建一次性配齐、也不要引入真实外部模型调用；只把 1 条最小回归验证跑通。

## 验收标准
1. 已新增并能运行 1 条 `pytest` 用例。
2. 该用例覆盖了当前最小任务入口，而不是纯粹测试一个无关工具函数。
3. 断言里明确检查了返回结构中的关键字段或字段命名。
4. README 或对应练习说明已同步 1 处调用示例、返回格式说明或测试运行方式。
5. 整体工作量控制在 20 到 60 分钟，没有扩展成“大补测试体系”。
