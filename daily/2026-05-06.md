# 今日任务：补空输入和异常场景处理

日期：2026-05-06

## 今日目标
让当前的 `/task` 或等价入口在遇到空输入、缺字段、非法参数或工具执行异常时，能返回清晰、稳定、可排查的结果，而不是直接报一团看不懂的错误。

## 为什么做这个
前几天你已经把 FastAPI 接口、Pydantic 模型、最小 tool schema 和最小 tool-calling flow 串起来了。下一步不该急着加更多能力，而是先把边界补稳。真实 Agent 服务里，最常见的问题不是“主流程写不出来”，而是“输入一脏就崩”“报错信息没人能看懂”“后面测试没法写”。把这一层补上，后面的 pytest、日志排查和 README 说明都会顺很多。即使你前一天还没完全做完最小工具流，今天这项任务也仍然成立，因为任何入口迟早都要处理坏输入和异常。

## 任务说明
1. 先盘点你当前 `/task` 或等价入口已经接收哪些输入字段，重点找 2 到 4 个最可能出问题的场景，例如空字符串、缺少 `text`、`top_k` 类型不对、工具层抛异常。
2. 用你现有的 Pydantic 模型、参数校验或显式分支，把这些坏输入挡在清晰的位置。目标不是“防住所有情况”，而是先把最常见的坑补住。
3. 约定一种统一、容易读的错误响应格式。可以很简单，但至少要让人看得出是哪个输入错了，还是工具执行过程出了问题。
4. 如果当前项目里已经有 logging，补 1 到 2 条关键日志，至少能区分“请求校验失败”和“工具调用失败”这两类问题；如果还没有成型日志，就先补最小可读日志，不要过度设计。
5. 补至少 1 个 pytest，用来验证一个异常或非法输入场景。时间够的话，再补 1 个正常场景，确保你没有因为加校验把主流程弄坏。
6. 顺手把 README 里和 `/task` 使用方式相关的部分看一眼。如果错误输入的行为已经比之前更明确，可以补 2 到 4 行说明，但今天不要求大改 README。
7. 不要在今天引入复杂异常体系、中间件重构、全局监控方案或新的外部依赖。目标是把当前最小服务的边界补稳，而不是做架构升级。

## 验收标准
1. `/task` 或等价入口至少能明确处理 2 类异常或非法输入场景。
2. 错误返回是清晰的，能区分输入问题和执行问题，而不是直接抛原始异常。
3. 至少有 1 条日志能帮助你排查失败发生在请求校验阶段还是工具执行阶段。
4. 至少有 1 个 pytest 覆盖了今天新增的异常或边界场景。
5. 整体工作量控制在 20 到 60 分钟，不追求一次性把异常处理体系做全。
