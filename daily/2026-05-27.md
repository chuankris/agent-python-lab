# 今日任务：给 `POST /task` 补 2 条最小 pytest

日期：2026-05-27

## 今日目标
围绕现有的 `POST /task`，补 2 条最小但有区分度的 `pytest`：1 条验证成功路径，1 条验证失败路径，让这个最小 Agent 接口开始具备“改了还能放心跑一下”的基础保护。

## 为什么做这个
如果前两天你已经把 `/task`、Pydantic 模型和最小 tool 流程接起来了，下一步最自然的工作不是继续堆功能，而是先把行为钉住。对 Agent 工程来说，很多回归不是出在“不会调模型”，而是出在改了输入结构、返回格式或错误处理之后，没有一个快速反馈。今天这一步很小，但很关键：它会把你从“接口能跑”推进到“接口开始可维护”。即使你前一天的输入校验还没完全补齐，今天也依然能做，因为你可以先针对当前已经存在的成功/失败行为写测试，再逐步收紧规则。

## 任务说明
1. 只围绕 `POST /task` 写测试，不要同时铺开到多个接口。
2. 先确定 1 条成功场景。建议用“合法输入进入 `/task` 后，返回固定结构”这一类最容易稳定的路径。
3. 再确定 1 条失败场景。优先选最明确的一条边界，例如文本为空、缺少必填字段，或 tool 名称不合法。
4. 这 2 条测试都要检查“结构”，不要只检查状态码。至少看清楚返回里是否包含你当前约定的关键字段。
5. 如果你已经有 Pydantic 响应模型，测试里优先按字段语义断言；如果还没有，就先把当前返回结构写清楚并固定住。
6. 测试命名尽量直接表达意图，例如“成功时返回工具结果”“非法输入时返回清晰错误”这一类，不要写成模糊名字。
7. 最后补 3 到 6 行 README 或相关说明，写清楚现在可以用什么命令跑这 2 条测试，以及这 2 条测试分别保护什么。

## 验收标准
1. 已新增并能运行 2 条围绕 `POST /task` 的 `pytest`。
2. 其中 1 条覆盖成功路径，1 条覆盖失败路径，而不是两条都只测 happy path。
3. 测试断言不只停留在 HTTP 200/422 之类的状态码，还检查了返回体中的关键字段或错误信息结构。
4. README 或相关说明里已经写明最小测试运行方式。
5. 今天没有继续扩展新功能，而是把当前最小闭环的行为固定住。
