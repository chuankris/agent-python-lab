# 今日任务：写第一个最小 tool schema，并补 1 条说明或校验

日期：2026-05-22

## 今日目标
为仓库补出第一个最小可读的 tool schema，先把“工具长什么样、接收什么参数、返回什么意图”描述清楚；如果时间够，再补 1 条最小校验或说明，让它能成为后面接 `/task` 和最小 tool-calling flow 的稳定起点。

## 为什么做这个
前几天你已经围绕 `POST /task` 逐步补了输入、模型、日志和测试思路，下一步最自然的不是立刻接真实模型调用，而是先把“工具契约”单独定义出来。Agent 工程里，tool schema 是模型和业务逻辑之间最小、最关键的边界：没有它，后面的调用流程容易越来越乱；先把它静态写清楚，后面无论你是手动调用、假调用，还是真接模型，都会轻很多。就算你前一天没完成，这个任务也依然能独立成立，因为它可以先作为一个单独模块存在，不依赖完整 `/task` 闭环已经完成。

## 任务说明
1. 先只选 1 个最小工具，不要贪多，建议用 `extract_keywords`、`summarize_text` 或 `classify_task` 这种一眼能理解的名字。
2. 用你当前最容易维护的方式定义 schema；可以先用 Python 字典，也可以用单独模块封装，但重点是字段命名清楚、结构稳定，不要追求花哨抽象。
3. schema 至少写清 4 件事：工具名、工具用途、输入参数结构、哪些参数必填。今天不要扩展到多个工具注册表。
4. 输入参数建议只保留当前真正需要的最小字段，例如 `text`；如果你想加 `max_keywords` 这类可选参数，也要控制在 1 个以内，避免把任务做大。
5. 如果仓库里已经有 `/task` 路由，可以先只在注释、README 或伪流程里说明“后面这里会调用这个 tool schema”；今天不要实现真实模型调用，不要接外部 API。
6. 补 1 条最小保护措施，二选一即可：要么写 1 条针对 schema 结构的 `pytest`，要么在 README 里补 3 到 6 行说明，写清这个工具接收什么输入、预期输出什么结果。
7. 如果时间还有余量，再顺手统一一下这个 schema 文件的格式，例如补类型提示、整理常量命名或跑一次格式化；但不要把今天扩展成完整 lint/重构任务。

## 验收标准
1. 仓库里已经有 1 个可单独阅读的最小 tool schema 文件，而不是只存在于临时想法里。
2. schema 至少包含工具名、用途、参数结构和必填字段这几项关键信息。
3. 今天没有实现真实模型调用，也没有把任务扩展成多工具编排，范围仍控制在 20 到 60 分钟内。
4. 至少补了 1 条最小说明或 1 条最小校验，用来固定这个 schema 的当前约定。
5. 你能用一句话解释：后面 `/task` 未来要怎样用到这个 schema。
