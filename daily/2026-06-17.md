# 今日任务：实现第一个可访问的健康检查接口

日期：2026-06-17

## 今日目标
在你的最小 FastAPI 练习里实现 `GET /health`，让服务第一次具备“启动后可访问、可验证”的真实接口，并顺手补 1 个最小 `pytest` 检查。

## 为什么做这个
路线图里，项目骨架之后最自然的一步就是健康检查接口。它很小，但能把几件关键基础串起来：FastAPI 路由、返回结构、启动命令、测试入口。

对 Agent 工程来说，`/health` 不是可有可无的示例接口，而是后面接日志、接 `/task`、接 tool schema 之前最便宜的一次联调点。先把这个点做实，后面每往前走一步都更稳。

如果你昨天的骨架还没完全落地，也没关系。今天可以先用 5 到 10 分钟把最小目录补齐，再继续做这个接口，本任务仍然成立。

## 任务说明
1. 进入你当前的最小练习目录，优先用 `exercises/day-001-project-skeleton/`；如果你昨天没有实际建出来，就先在这个目录下补出最小 `app/`、`tests/` 和 `README.md`。
2. 在 FastAPI 入口文件里增加 `GET /health` 路由，返回固定结构，建议先保持最小版本，例如只返回服务状态，不要顺手扩展版本号、时间戳或复杂字段。
3. 如果你已经开始使用 Pydantic，可以思考这个返回值后面是否值得收敛成响应模型；但今天不强制要求你把 health 响应单独抽成模型文件。
4. 补 1 个最小 `pytest` 用例，验证 `GET /health` 能返回 `200`，并且响应体里至少包含一个你约定的关键字段。
5. 检查 README，把启动命令和这条接口的访问示例补上；示例可以是浏览器访问方式，也可以是 `curl`。
6. 如果还有时间，再看一眼返回格式是否够稳定，尽量用统一的小写字段名和清晰的 JSON 结构，但不要继续扩成第二个接口。
7. 今天不要做 `/task`、不要接模型、不要做真实 tool calling；重点是把“服务启动 -> 访问接口 -> 跑测试”这条最小链路打通。

## 验收标准
1. 本地已经存在一个可启动的 FastAPI 应用，并且有明确入口文件。
2. `GET /health` 可访问，返回固定 JSON 结构。
3. 至少有 1 个最小 `pytest` 用例覆盖这条接口的成功场景。
4. README 至少补充了启动命令和 `GET /health` 的访问示例。
5. 整体工作量控制在 20 到 60 分钟，没有顺手扩写 `/version`、`/task`、日志体系或额外工具能力。
