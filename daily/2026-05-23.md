# 今日任务：整理最小 tool-calling 流程说明，并补 1 处轻量收口

日期：2026-05-23

## 今日目标
把仓库里“`/task` 将来如何接上 tool schema”的最小流程写清楚，并顺手补 1 处轻量收口动作：二选一完成 README 说明，或整理 1 个现有 schema/测试文件的格式与命名。

## 为什么做这个
昨天如果你已经写了第一个最小 tool schema，今天最自然的下一步并不一定是立刻接真实模型调用，而是先把流程边界说清楚：请求进来以后，哪里决定要不要用工具，哪里读取 schema，哪里返回结果。对 Agent 工程来说，这种“先把边界讲明白”的能力很重要，因为后面接模型、日志和异常处理时，混乱通常不是出在代码不会写，而是出在职责没有提前画清。今天又是周末，适合做一次轻量收口，把前两天的成果整理得更稳一些。就算你昨天没完成，这个任务也依然能独立做，因为你可以先按预期结构补说明，再反过来约束后续实现。

## 任务说明
1. 先用你自己的话定义一个最小闭环：`POST /task` 接收输入，判断是否需要某个工具，找到对应 schema，执行一个“假调用”或本地占位逻辑，最后返回结构化结果。
2. 今天不要接真实大模型，不要接外部 API，也不要设计多工具编排；只描述或整理 1 个工具的最小流转。
3. 把这段流程写到 README，或写到与工具相关的说明位置，控制在 6 到 12 行内，重点写清“输入 -> 判断 -> 工具 -> 输出”。
4. 如果 README 里已经有相关内容，就改成更具体的版本，避免只写“后续会支持工具调用”这种空话。
5. 再补 1 处轻量收口动作，二选一即可：
   - 给现有 tool schema 文件补更清楚的字段命名、注释或格式整理；
   - 给已有测试文件补一个更准确的测试名，或把重复断言整理得更易读。
6. 如果你仓库里还没有真正的工具执行函数，也没关系；今天允许只用“伪流程”或“占位函数”来表达设计，不要求写完整实现。
7. 完成后自己检查一次：一个刚加入项目的人，看这段说明能不能在 1 分钟内理解你准备怎么把 `/task` 接到工具上。

## 验收标准
1. 今天至少新增或改动 1 处与“最小 tool-calling 流程”直接相关的文字说明。
2. 说明里明确出现输入、判断、工具/schema、输出这 4 个环节中的大部分，而不是泛泛描述。
3. 额外完成了 1 处轻量收口动作：README 更具体，或 schema/测试文件更整洁、更好读。
4. 没有扩展到真实模型接入、多工具注册表或复杂重构，整体仍能控制在 20 到 60 分钟内。
5. 你能用一句话解释：如果明天开始真写最小 tool-calling，第一刀会改哪个文件。
