# 今日任务：给最小 `/task` 流程补一条结构化日志

日期：2026-06-11

## 今日目标
让你当前的最小 FastAPI `/task` 流程在“成功请求”这条路径上多一层可观察性：补 1 条结构化日志，补 1 个对应的 `pytest` 用例，顺手把 README 里的日志或运行说明对齐。

## 为什么做这个
前几天你已经围着 `/task`、Pydantic、tool schema、最小响应结构、README 在收口。下一步不该急着加更复杂的 Agent 能力，而是先让这条最小链路“更容易看清楚”。  
对 Agent 工程来说，日志不是附属品。后面无论你接模型、接工具、补异常处理还是做评测，第一件事几乎都是先看这次请求进来了什么、走到了哪一步、最后产出了什么。今天这一步很小，但它能把 FastAPI、日志、测试、文档这四块再拧紧一点。  
如果你前一天还没做完响应结构收口，也没关系：今天可以基于你现在真实存在的 `/task` 成功响应做日志，不要求你先把整套契约重构完。

## 任务说明
1. 先确认你当前 `/task` 成功路径里，哪些字段已经稳定可用；只围绕这些真实字段打日志，不要为了写日志临时发明新结构。
2. 在成功请求完成后补 1 条结构化日志，优先记录 3 类信息：请求的核心输入、命中的工具名或当前阶段、响应里的 1 到 2 个关键字段。
3. 日志内容控制在“排查问题时够用”即可，不要把整段原始输入、调试细节、内部对象一次性全打出来。
4. 如果你还没有正式的 logging 配置，今天只补最小可运行方案即可；重点是先形成一致的日志出口，而不是把完整可观测体系一次做完。
5. 补 1 个 `pytest` 用例，验证成功请求时这条路径仍然可跑；如果你会测日志输出，就断言日志里至少包含 1 个核心字段；如果暂时不会测日志，就至少断言补日志后接口行为没有被改坏。
6. README 里补 1 小段说明：当前项目把 `/task` 成功路径的关键信息记到哪里，或者本地如何看到这条日志。
7. 如果你发现自己其实还没有 `/task`、`app/` 或 `tests/`，不要在今天硬补整套实现；把任务降级成“先给最小练习骨架准备 logging 入口和测试占位”，工作量仍然控制在 20 到 60 分钟。

## 验收标准
1. `/task` 成功路径新增了 1 条实际会输出的结构化日志。
2. 这条日志至少包含请求核心输入、工具名或当前阶段、响应关键字段中的其中 2 类信息。
3. 至少新增或更新 1 个 `pytest` 用例，验证补日志后最小流程仍然正常。
4. README 或练习说明里补了 1 处与日志查看方式相关的说明。
5. 整体改动仍然是“小收口”，没有扩展成复杂日志框架、评测系统或多工具编排。
