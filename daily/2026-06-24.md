# 今日任务：给 `/task` 接上第一个本地工具骨架

日期：2026-06-24

## 今日目标
在你当前的最小 FastAPI 练习里，为 `POST /task` 接上一个“不依赖 LLM”的本地工具骨架：先定义 1 个最小 tool schema，再让接口走通一次“输入 -> 选择工具 -> 返回结构化结果”的占位流程。

## 为什么做这个
你前面几天已经在围绕 FastAPI、Pydantic、最小测试和日志打地基。下一步最自然的推进，不是立刻接真实模型，也不是一下子把 MCP、复杂编排和多工具调度全塞进去，而是先把 Agent 工程里最核心的一段骨架跑通：接口收到请求后，能按固定规则调用一个本地工具函数，并返回可预测结果。

这一步对你从 Java/IoT 集成转过来尤其关键。它很像你以前先把“请求协议、处理入口、下游调用边界、回包结构”钉住，再逐步把真实能力替换进去。今天先做一个极小的本地版 tool-calling flow，后面你再接真实 LLM、补异常处理、扩展 pytest、整理 README，整体会顺很多。

## 任务说明
1. 继续沿用你已经在做的练习目录，优先使用 `exercises/day-001-project-skeleton/`，不要新开第二套项目。
2. 如果你还没有可运行的 `POST /task`，先补一个最小占位版本：能接收请求体并返回固定结构即可；补到能继续今天任务就停，不要顺手扩展成复杂业务。
3. 选一个足够小的本地工具主题，推荐 `extract_keywords`。它今天不需要“聪明”，甚至可以先只做最朴素的规则版实现，比如按空格切分、去重、截前几个词，重点是先有“工具输入/输出边界”。
4. 先定义这个工具的最小 schema。你可以用最简单的 Python 字典，也可以用现有的 Pydantic 模型承载，但今天至少要把下面三件事说清楚并落到代码结构里：
   - 工具名是什么
   - 工具需要什么输入
   - 工具返回什么结构
5. 在 `/task` 里增加一层非常轻的分发逻辑：先接收用户文本，再调用你这个本地工具，最后把工具结果包进接口响应里。今天不要接真实 LLM，不要做动态工具选择，也不要引入多工具注册系统。
6. 响应结构尽量保持克制，推荐只多加 2 到 3 个字段，例如 `tool_name`、`result`、`status`。重点是“能看出这次请求已经走过一次工具调用”，不是字段越多越好。
7. 至少补 1 个 `pytest` 用例，验证 `/task` 成功调用本地工具后的返回结构。最低要求是检查状态码、工具名和结果字段存在；如果你还有余量，再补 1 个针对空输入或短输入的简单断言，但不要把今天扩展成完整异常处理日。
8. 在练习目录的 `README.md` 里补 4 到 8 行说明，写清楚：
   - 今天新增了哪个本地工具
   - `/task` 现在的最小处理流是什么
   - 为什么今天故意不接真实 LLM，只做本地占位工具流
9. 如果你还有大约 10 分钟余量，只做一个很小的收口动作即可：统一 schema 命名、把工具函数单独放进一个更清楚的文件，或者把测试名改得更直白。不要继续扩成“可配置工具框架”。

## 验收标准
1. 本地已经存在一个可运行的 `POST /task`，并且它会实际调用 1 个本地工具函数，而不是只返回固定假数据。
2. 这个本地工具已经有清晰的最小 schema 或等价结构，能说明输入和输出长什么样。
3. `/task` 的响应里能明确看出本次请求使用了哪个工具，以及工具返回了什么最小结果。
4. 至少有 1 个 `pytest` 用例覆盖 `/task` 成功走通本地工具调用的场景。
5. `README.md` 已补充今天这条最小 tool-calling flow 的说明，并明确当前范围仍然是本地占位实现，不是真实 Agent 编排。
6. 整体工作量控制在 20 到 60 分钟，没有顺手扩展到真实 LLM 接入、多工具系统、复杂异常处理或完整框架化重构。
