# Day 005 / 2026-04-30

## 今日目标
把现有 `POST /task` 接口的输入和输出收口成清晰的 Pydantic 模型，让接口从“能跑”升级到“有明确数据边界”。

## 为什么做这个
昨天你已经把最小的任务接口链路搭起来了，今天该补的是数据结构这层基本功。对 Agent 服务来说，请求体、响应体、字段命名和校验规则越早定清楚，后面接日志、测试、tool schema 和最小 tool-calling 流程时越不容易乱。这个任务也很适合补做：即使你昨天没完全完成，也可以直接从一个最小 `POST /task` 出发推进。

## 任务说明
1. 为 `POST /task` 明确拆出一个请求模型和一个响应模型，使用 Pydantic 来表达字段结构。
2. 请求模型至少包含一段任务文本；响应模型至少包含原始输入内容，以及一个你定义的状态字段或结果字段。
3. 给其中一两个字段补最小但明确的约束，例如必填、默认值、字段说明，或者你认为合理的简单长度限制，但不要把校验规则设计得过重。
4. 把路由函数改成直接依赖这些模型，而不是在函数体里手写松散的字典处理。
5. 如果 `README.md` 还没写清楚 `POST /task` 的请求示例和返回示例，就补一小段，帮助你自己下次快速回忆这个接口怎么验证。

注意：
- 今天重点是“模型边界清楚”，不是扩展业务逻辑。
- 不要接入真实模型调用。
- 不要顺手把日志、pytest、tool schema 一次性做完，后面几天会继续拆开推进。

## 验收标准
1. `POST /task` 使用了明确的 Pydantic 请求模型和响应模型。
2. 接口仍然可以本地启动并正常返回结构化 JSON。
3. 至少有一个字段体现了最小校验或默认值设计，而不只是裸类型声明。
4. `README.md` 至少补清楚一种 `POST /task` 的请求示例或返回示例。
5. 整体改动控制在 20 到 60 分钟内可完成。
