# 今日任务：给项目补上最小格式化和 lint 入口

日期：2026-05-08

## 今日目标
为当前项目接入一个最小可用的代码格式化和 lint 入口，优先选择 `ruff`，让你后面继续改 FastAPI、Pydantic 模型、日志、pytest 和 tool schema 时，能先用一条命令做基础质量检查。

## 为什么做这个
前几天你已经逐步碰到了接口、模型、日志、异常和测试。下一步不是继续堆功能，而是先把“代码改完后怎么快速自检”补起来。对 Agent 工程来说，格式化和 lint 的价值不只是代码好看，它能帮你尽早暴露导入混乱、未使用变量、基础风格不一致这些低级问题，减少后面重构 `/task`、补工具流和整理 README 时的噪音。即使你前一天的测试还没补完，今天这件事也不冲突，因为它本质上是在给后续每天的小步提交加一个稳定护栏。

## 任务说明
1. 先确认项目里有没有现成的 Python 配置文件，例如 `pyproject.toml`、`requirements.txt` 或 `README.md` 里的运行说明；如果还没有，就以最小可运行为目标，不要为了“标准化”一次性补全整套工程配置。
2. 优先接入 `ruff` 作为今天的唯一主角。如果你已经熟悉 `black`，也可以只做“`ruff check` + `ruff format`”这一套，不必同时上很多工具。
3. 给项目补一个清晰的执行入口。形式可以是 `pyproject.toml` 里的配置、README 里的两条命令，或者一个非常轻量的脚本说明；重点是以后你自己能一眼知道“改完代码先跑什么”。
4. 先把检查范围控制在当前真实会动的目录，例如 `app/`、`tests/`，如果现在还没有这些目录，也可以先对未来会放代码的位置写清约定，不要把今天做成空转配置秀。
5. 如果你手头已经有几段 Python 代码，就实际跑一次格式化或 lint，顺手修掉 1 到 3 个最明显的小问题；如果当前仓库还几乎没有代码，就把重点放在配置最小入口和 README 说明，不要硬凑规则。
6. README 里补 3 到 6 行最小说明，至少写清安装后怎么执行格式化检查、怎么执行自动格式化，避免过两天你自己都忘了命令。
7. 不要在今天引入 pre-commit、CI 工作流、复杂规则分层或大规模目录重构。先把“本地一条命令能自检”立住就够了。
8. 如果时间还有余力，再顺手检查一下昨天的测试文件命名和接口模块导入，看看新接入的 lint 是否能帮你提前发现一个真实问题，但这不是今天的硬性要求。

## 验收标准
1. 项目里已经有一个明确的格式化和 lint 入口，优先是 `ruff`。
2. 你能用一条或两条固定命令对当前项目执行基础检查或自动格式化。
3. README 或等价说明里已经写清这些命令怎么跑。
4. 如果仓库里已有 Python 代码，至少实际跑过一次检查，并处理了最明显的 1 类问题；如果还没有代码，也完成了最小配置和使用说明。
5. 整体工作量控制在 20 到 60 分钟，不扩展到 CI、pre-commit 或复杂工程治理。

- `tests/`
