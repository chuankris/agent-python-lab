# 今日任务：补一组最小回归检查并对齐说明

日期：2026-06-13

## 今日目标
围绕你当前已经存在的最小 FastAPI/Agent 练习，补 2 个以内的小检查点：优先补 `pytest` 用例，其次同步 1 处 README 或练习说明里的命令/示例，让 `/task` 这条最小链路更稳一点，但不新增功能。

## 为什么做这个
这周你已经围绕 `/task`、Pydantic、最小 tool schema、响应结构和日志做了连续收口。到周末，更适合做一次轻量巩固，而不是再加新能力。

对 Agent 工程来说，真正容易掉链子的地方，往往不是“功能不会写”，而是测试没补、示例过期、命令不一致、下次打开项目时不知道该先跑什么。今天这条任务就是把这些小裂缝补上，让你练到“能持续维护最小服务”，不只是“写出一个接口”。

如果你前两天有任务没做完，也没关系。今天的任务可以直接基于你仓库里现在真实存在的接口和命令来做，不要求你先把整条 `/task` 链路补齐。

## 任务说明
1. 先确认当前仓库里哪条最小链路是真正可运行的，优先看 `POST /task`；如果 `/task` 还不稳定，就退一步选择你已经能跑通的最小接口。
2. 补 1 到 2 个 `pytest` 检查点，优先顺序是：成功请求、无效输入、日志或响应关键字段三选二，不要一口气把测试体系铺大。
3. 如果你已经有 `tests/`，就在现有风格上补；如果还没有成型测试结构，只补最小可运行用例，不顺手重构整个目录。
4. 检查 README 或对应练习目录里的 README，把其中 1 处最容易过期的内容同步到当前真实状态，例如启动命令、测试命令、`/task` 请求示例、返回字段名、格式化命令。
5. 如果项目里已经接入 `ruff`、`black` 或等价工具，今天至少实际跑 1 条格式化或 lint 命令；如果还没接，就只把 README 里的命令说明补准确，不要求今天新装工具。
6. 今天不要扩展新接口、不要新增第二个 tool、不要引入数据库或异步编排；重点是让已有最小链路更可验证、更好接手。
7. 完成后自己写一次简短 checkin，记录“补了哪两个检查点、同步了哪一处说明、还有哪一处暂时没收口”。

## 验收标准
1. 至少新增或更新了 1 个可运行的 `pytest` 用例。
2. 如果时间够，最多再补 1 个小检查点，总量控制在 2 个以内，没有演变成大面积补测试。
3. README 或练习说明中至少有 1 处内容和当前真实代码重新对齐。
4. 至少确认了 1 条测试命令，且如果仓库已有格式化/lint 工具，也确认了 1 条对应命令。
5. 整体工作量控制在 20 到 60 分钟，没有写新功能，也没有直接照着答案堆实现代码。
