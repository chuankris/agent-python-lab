# 今日任务：为 `/task` 补 2 到 3 条最小 pytest 用例

日期：2026-06-06

## 今日目标
围绕你现在的 `/task` 或等价入口，补 2 到 3 条最小 `pytest` 用例，优先覆盖 1 个正常场景和 1 到 2 个非法输入场景，让这条最小 tool-calling flow 有一层稳定的回归保护。

## 为什么做这个
你这几天已经把 tool schema、最小调用闭环、空输入与异常返回逐步补起来了。周末不适合再往前推新功能，更值得做的是把这条链路用测试钉住。对 Agent 服务来说，很多问题不是“写不出来”，而是后面一改字段名、返回结构或校验逻辑就悄悄退化。今天这一步很小，但会直接训练你用 `pytest` 描述接口行为、用断言固定响应结构，也能顺手检查 README 里的示例是不是已经和当前实现一致。即使你前一天没完全做完，这个任务也能独立推进，因为你只需要基于当前能运行的入口补最小验证。

## 任务说明
1. 先确定你现在用于提交任务的入口是什么，优先还是 `POST /task`；如果还没正式挂成路由，就围绕等价处理函数写测试。
2. 只补 2 到 3 条测试，不要一口气铺很多 case。推荐组合是：1 条正常请求、1 条缺少字段、1 条空字符串或空白字符串。
3. 正常场景的断言重点放在“响应结构稳定”上，例如状态码、是否包含主要结果字段、是否保留工具调用后的关键输出。
4. 非法输入场景的断言重点放在“错误可读”上，例如状态码明确、错误字段命名一致、关键提示信息看得懂。
5. 如果你前几天的字段命名还不完全统一，今天顺手把测试里的命名先定下来，再检查 README 或示例请求是否需要一起对齐 1 处。
6. 测试数据保持最小，不要引入外部模型、真实网络请求或复杂 fixture；能直接构造请求就不要过度抽象。
7. 今天不要扩展到参数化测试矩阵、覆盖率统计、CI 配置或完整测试分层；只把当前最小主链路测清楚。

## 验收标准
1. 已新增 2 到 3 条 `pytest` 用例，覆盖至少 1 个正常场景和 1 个非法输入场景。
2. 正常场景测试能说明 `/task` 当前成功响应的最小结构。
3. 非法输入测试能说明当前状态码和错误提示是清楚、稳定的。
4. README、练习说明或示例请求中，至少有 1 处内容与测试里使用的字段命名保持一致。
5. 整体工作量控制在 20 到 60 分钟，没有扩展成完整测试体系改造。
