# 今日任务：给 `POST /task` 补最小结构化日志，并记录 1 次手动验证

日期：2026-05-18

## 今日目标
为当前最小 `POST /task` 接口补一层够用的结构化日志，让你在本地发起 1 次请求时，能明确看到“请求进来了、处理结束了、结果是成功还是失败”。

## 为什么做这个
前两天你已经把 `/task` 往“有模型、有测试、有 README 说明”的方向推进了一步，下一步最自然的不是立刻接复杂 tool-calling，而是先把可观察性补上。对 Agent 服务来说，日志不是锦上添花，而是后面排查输入问题、工具调用问题和异常场景的最小抓手。即使你前一天没完成，今天也依然成立：只要有一个最小 `/task` 路由，就可以先把日志链路搭起来。

## 任务说明
1. 如果仓库里还没有稳定可访问的 `POST /task`，先补到“能接收最小请求并返回结果”为止，再继续今天任务。
2. 先确定今天只记录哪些最小信息，建议控制在 3 到 5 个点以内，例如接口名、请求是否成功、输入文本长度、返回状态码、错误摘要。
3. 给 `/task` 的正常路径补 1 条开始日志和 1 条结束日志；不要一上来铺太多字段，先把最小闭环打通。
4. 如果你已经有无效输入或异常分支，再额外补 1 条失败日志，但今天只需要覆盖 1 种失败情况，不要扩展成完整异常体系。
5. 日志内容尽量保持结构稳定，避免整段自由文本；重点是让你之后一眼能 grep 到关键信息。
6. 手动发 1 次请求，确认终端里确实能看到这组日志，并把你观察到的现象补 2 到 4 行到 README 或相关说明里。
7. 今天不要引入第三方日志平台、不要加 request id 中间件、不要实现 tool schema；把“最小可观察”这一小步做好就够了。

## 验收标准
1. 本地 `POST /task` 在收到请求时，至少会输出 1 条开始日志和 1 条结束日志。
2. 日志里至少包含 2 个稳定字段，而不是只有随意拼接的一句文本。
3. 如果仓库里已经有失败输入场景，至少有 1 种失败情况会输出清晰日志；如果还没有，也可以只先完成成功路径。
4. README 或相关说明里，已经写下你如何触发这次验证，以及终端日志大致能看到什么信息。
5. 今天的改动仍控制在 20 到 60 分钟内，没有扩展到完整异常体系、真实 tool-calling 或复杂可观测平台。
