# 今日任务：为最小 task 流程补 1 个 pytest 场景并同步 README 示例

日期：2026-05-31

## 今日目标
围绕你当前最小的 FastAPI / Agent 练习，补 1 个能跑通的 `pytest` 场景，顺手把 README 里的请求或返回示例同步到当前实现计划。

## 为什么做这个
这一阶段你已经连续碰到 FastAPI、Pydantic、logging、tool schema、最小 tool-calling flow、格式化和 README。周末不适合继续加新功能，更适合把“最小链路是否可验证”这件事压实。很多 Agent 小项目后面越改越乱，不是因为功能太难，而是因为没有尽早固定“接口长什么样、测试怎么验证、文档怎么跟上”。今天这一步很轻，但很像真实工程里的周末收口：不扩功能，只补最小验证和说明。即使你前一天没做完，也可以独立完成，因为你只需要围绕当前手里最接近 `/task` 的那条链路工作。

## 任务说明
1. 先选 1 条你现在最接近完成的最小链路，优先是 `POST /task`；如果你还没写到这里，就选一个等价的最小入口，例如准备中的请求模型加返回结构。
2. 明确这条链路里最值得先锁定的 1 个行为，例如“合法请求能返回固定字段”“缺少必填字段会报错”“tool schema 里的某个字段会出现在返回里”。
3. 只补 1 个 `pytest` 场景，不追求覆盖率。重点是把输入、断言字段、预期状态码或预期错误写清楚。
4. 如果你已经有 Pydantic 模型，测试断言尽量对齐模型字段名；如果你已经有 tool schema，测试里至少出现 1 个 schema 相关字段，避免测试和接口各说各话。
5. 打开当前练习目录下的 README，把与这条链路相关的 1 段示例同步掉。可以是请求体示例，也可以是返回示例，但必须和你今天验证的场景一致。
6. 如果你还没有真正实现接口，也可以先写“红灯测试”或先整理测试输入输出草稿，但不要在今天把任务扩成完整功能开发。
7. 今天不要新增第二个接口，不要补一串测试，不要引入更复杂的 tool 调度；把 1 条最小链路的验证和说明收口即可。

## 验收标准
1. 已新增或补齐 1 个最小 `pytest` 场景。
2. 这个场景明确验证了 1 个接口行为、字段约束或 tool schema 相关约束。
3. README 中至少有 1 处请求或返回示例与今天的测试目标保持一致。
4. 今天的改动仍然控制在 20 到 60 分钟内，没有扩成新功能开发。
5. 你能用一句话说明：当前这条最小 task 流程，先用哪条测试来确认它没跑偏。
