# 今日任务：先为 `/task` 立最小输入输出契约

日期：2026-06-21

## 今日目标
在你当前的最小 FastAPI 练习里，先为后续的 `POST /task` 准备一套最小请求/响应契约：补 2 个 Pydantic 模型、给接口加一个最小占位实现，外加 1 个最小 `pytest` 和 1 段 README 调用示例。

## 为什么做这个
你前两天已经在做“接口能跑起来”和“最小测试能跟上”的基本功。下一步最自然的推进，不是立刻把 Agent、tool schema、日志和调用流程一口气全塞进去，而是先把 `/task` 这条主线接口的输入输出结构收紧。

这一步很像你以前做 Java/IoT 集成时先定请求体、响应体和错误边界。放到 Python/FastAPI 这里，本质没变，只是你要开始用 Pydantic 把契约写得更轻、更直接。后面无论你是接 logging、补异常处理、还是把 tool schema 和最小 tool-calling flow 接上，这两个模型都会继续复用。

今天是周日，任务刻意收轻一点：先把“结构”定住，不追求真实 Agent 能力，不追求复杂业务逻辑。只要把 `/task` 的骨架搭起来，明天继续往里填就顺了。

## 任务说明
1. 继续在你已经使用的练习目录里做，优先沿用 `exercises/day-001-project-skeleton/`，不要新开第二套项目。
2. 为即将引入的 `POST /task` 先定义最小请求模型和响应模型。请求体建议只保留 1 个核心字段，比如 `text`；响应体建议只保留 2 个字段以内，比如 `task_id` 和 `status`，或者 `input_text` 和 `accepted`。重点是“结构清楚”，不是字段多。
3. 在 FastAPI 入口里加上 `POST /task` 路由，但今天只做最小占位实现即可。它可以先接收请求模型，再返回一个固定的、可预测的响应模型；不要在今天引入真实 LLM 调用、真实 tool schema、数据库、消息队列或异步任务。
4. 如果你现在把模型、路由都写在同一个文件里，今天允许做一次很小的整理：把 Pydantic 模型抽到 `schemas.py`。但不要顺手做大重构。
5. 补 1 个最小 `pytest` 用例，验证 `POST /task` 的成功场景。测试至少检查状态码和响应 JSON 的关键字段，不要求今天把异常场景一起补完。
6. 在练习目录的 `README.md` 里补一段最小调用示例，写清楚请求 JSON 长什么样、响应 JSON 大致长什么样，并用一句话说明“今天只是在固定接口契约，后面再接真正的 Agent 流程”。
7. 如果你还有 10 分钟左右余量，再做一个很小的收口动作即可：统一字段命名风格，或者把测试文件名改得更清楚。不要继续扩到 logging、tool schema 执行、格式化器接入或复杂异常处理。

## 验收标准
1. 本地已经存在 `POST /task`，并且可以接收一个最小 JSON 请求体。
2. `POST /task` 已经使用 Pydantic 请求/响应模型，而不是直接收发裸字典。
3. 至少有 1 个 `pytest` 用例覆盖 `/task` 的成功场景。
4. `README.md` 已补上 `/task` 的最小调用示例，并明确今天的范围只是“固定契约和占位返回”。
5. 整体工作量控制在 20 到 60 分钟，没有把任务扩展成真实 tool-calling 或完整 Agent 服务。
