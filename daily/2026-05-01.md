# 今日任务：给 `/task` 补最小日志

日期：2026-05-01

## 今日目标
给现有的 FastAPI 接口补一层最小但清晰的日志，让你能看到一次请求是怎么进来、怎么返回的，先建立最基础的可观测性习惯。

## 为什么做这个
前几天你已经把服务、路由和 `POST /task` 的基本链路逐步搭起来了。接下来先不要急着接真实模型或复杂工具调用，先把日志这层补上。对 Agent 工程来说，日志不是附属品，而是后面排查输入问题、响应格式问题、tool 调用问题时最先依赖的现场信息。

## 任务说明
1. 在当前 FastAPI 项目里，为 `POST /task` 增加最小日志输出。
2. 日志至少覆盖两类信息：收到请求时记一条，返回结果前再记一条。
3. 日志内容尽量结构化，至少包含接口名、请求状态，以及一个你认为后面排查问题时会有用的字段，例如任务文本长度、状态字段值或请求 ID 占位。
4. 先不要引入复杂日志框架配置，重点是把标准库 `logging` 或你当前最容易跑通的最小方案接起来。
5. 如果你昨天的 `POST /task` 还没完全做完，今天也可以先在任意一个已存在接口上把日志链路跑通，再补到 `/task`，但最终最好能落在 `/task` 上。
6. 顺手在 `README.md` 补一小段说明：启动服务后，怎样通过控制台观察到这两条日志。

## 验收标准
1. 本地启动服务后，请求 `POST /task` 时，控制台能看到至少两条和这次请求有关的日志。
2. 日志不是随手打印一整段文本，而是能看出清晰字段或固定格式。
3. 接口原有返回不被破坏，仍然能正常返回 JSON。
4. `README.md` 至少补清楚一种查看日志的方式。
5. 整体改动控制在 20 到 60 分钟内可完成。

## 推荐目录/文件
- `app/main.py`
- `app/`
- `README.md`
- `tests/`（今天可以先不补测试，但要想一下明天怎么验证日志相关行为）
