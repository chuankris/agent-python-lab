# 今日任务：收口最小 Agent 服务 README

日期：2026-05-29

## 今日目标
把当前最小 FastAPI / Agent 练习的 README 收口成一份新人自己隔一天再回来，也能在几分钟内看懂并重新跑起来的说明。

## 为什么做这个
你这几天已经在围绕 FastAPI、Pydantic、日志、`pytest`、tool schema 和最小 tool-calling flow 往前推。这个阶段最容易出现的问题，不是功能不会写，而是过几天回头看时，不确定入口文件在哪、`/task` 接口收什么、测试和格式化命令先跑哪个。README 收口看起来不像“写功能”，但它直接决定你后面每次继续练习时的启动成本。即使你前一天还没把格式化完全补齐，今天也能做，因为重点是把现状写清楚，而不是扩新能力。

## 任务说明
1. 只整理当前最小练习真正已经存在的内容，不要在 README 里预支还没实现的功能。
2. 用 3 到 6 行先写清楚项目现在的最小目的，例如“这是一个带 `/task` 接口的最小 Agent 服务练习”，避免 README 一上来太空。
3. 单独补一个“快速开始”小节，至少写清楚 3 类命令：启动服务、运行测试、执行格式化；命令数量尽量少，能直接复制更重要。
4. 给 `/task` 或当前核心接口补一个最小示例，至少说明请求体里有哪些关键字段、返回里最值得先关注什么，不要求写很长，只要能帮助你自己下次快速回忆。
5. 如果仓库里已经有 Pydantic 模型、tool schema 或日志初始化代码，就在 README 里点名它们各自大概放在哪个目录；如果还没有完整拆分，也如实写“当前先集中在某个文件”，不要假装结构已经很完整。
6. 补 2 到 4 行“改动顺序建议”，例如先改 schema，再跑 `pytest`，最后跑格式化，让 README 不只是说明书，也能提示最小工作流。
7. 今天不要新增接口、不要重构目录、不要为了 README 去改一大批代码；目标只是把现在这套最小闭环说明清楚。

## 验收标准
1. README 或当前练习目录说明已经能让你快速回答“怎么启动、怎么测试、怎么格式化、核心接口在哪”。
2. 文档里至少包含 1 个最小请求或返回示例，而不是只有口头描述。
3. 文档内容与当前代码现状一致，没有把未实现能力写进去。
4. 至少有 1 处关于 Pydantic 模型、tool schema、日志或测试位置的说明被补充清楚。
5. 今天没有把任务扩成代码功能开发，而是把最小 Agent 服务的使用说明收口完成。
