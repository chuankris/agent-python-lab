# 今日任务：给 `POST /task` 补输入校验和错误返回

日期：2026-05-26

## 今日目标
让当前最小 `POST /task` 闭环更像一个真实服务：补 1 到 2 个清晰的输入校验规则，并在出错时返回稳定、可读的错误信息。

## 为什么做这个
昨天你已经把 `/task` 和 1 个本地占位工具接起来了。下一步最自然的工作不是继续加功能，而是先把边界收紧。Agent 服务一开始常见的问题不是“不会调用”，而是“输入一乱就不知道该怎么坏掉”。对有 Java 和集成经验的人来说，这一步很值得认真做，因为它直接决定接口能不能被别人稳定接入。即使你昨天还没完全接好工具，今天也照样能做：只要已经有 `POST /task`，就可以先把请求体和错误返回约定清楚。

## 任务说明
1. 先选 1 个最小但真实的输入约束，不要一次加很多规则。比如：
   - 文本字段不能为空；
   - 文本长度需要大于某个最小值；
   - 工具名如果存在，必须是当前允许的那个值。
2. 优先用 Pydantic 模型表达这些约束，而不是把校验散落在路由函数里。
3. 再补 1 个最小异常场景处理。可以是请求参数不合法，也可以是本地占位工具收到不支持输入时返回清晰错误。
4. 错误返回要能让人一眼看出问题，不要只留一个模糊的 `error` 字符串。至少保证字段名稳定、语义直接。
5. 如果你已经有日志，补 1 条最小错误日志；如果还没有日志基础，也可以先记录“本次请求为什么被拒绝”这一类关键信息。
6. 补 1 条 `pytest`，专门验证一个失败场景。今天不要追求覆盖全部异常，只固定住最重要的一条边界。
7. 最后看一眼 README 或接口说明，补 3 到 5 行，写清楚当前 `/task` 至少有哪些输入限制，调用者出错时会看到什么。

## 验收标准
1. `POST /task` 至少新增了 1 个明确输入约束，且这个约束能稳定触发。
2. 至少有 1 个失败场景会返回清晰、结构稳定的错误信息。
3. 至少补了 1 条针对失败场景的 `pytest`，而不是只测成功路径。
4. 如果已经有日志体系，能看到与本次失败场景对应的最小日志信息。
5. README 或相关说明里已经写清 1 条输入限制或 1 类错误返回约定。
