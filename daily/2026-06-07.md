# 今日任务：接入最小格式化检查并整理 1 处 README

日期：2026-06-07

## 今日目标
为你当前的最小 FastAPI/Agent 练习接入 1 套最小格式化或 lint 工具，优先选 `ruff` 或 `black` 其中一个，并顺手整理 README 里 1 处运行说明或目录说明。

## 为什么做这个
这几天你已经把 `/task`、Pydantic、最小 tool flow 和 `pytest` 慢慢搭起来了。周末更适合做一次轻量收束，而不是继续加新功能。对 Python 和 Agent 工程来说，格式化和基础 lint 不是“锦上添花”，而是后面继续迭代时最省心的底座：命名、导入、空行、示例命令如果一直漂，README、测试和接口说明很快就会开始互相打架。今天这一步很小，但能帮你把项目往“更像一个能持续维护的小服务”再推半步。即使你前一天没完全做完，这个任务也能独立推进，因为它主要是收紧工程习惯，不依赖复杂业务逻辑。

## 任务说明
1. 先选 1 个最小工具，优先推荐 `ruff`；如果你更熟悉，也可以先只接 `black`，今天不要同时折腾太多工具。
2. 只做最小接入：让项目里能明确看出“以后应该用什么命令检查或格式化代码”，不要求今天把整套开发流程全部配齐。
3. 如果仓库还没有相关配置，可以补 1 份最小配置；如果已经有配置，就只检查它是否和当前目录结构、命令写法一致。
4. 运行 1 次你选的格式化或检查命令，确认至少能作用到当前最小 FastAPI/练习代码目录，而不是只把工具装上但没人知道怎么用。
5. README 或对应练习说明里补 1 处最小文档，写清楚“本地如何执行格式化/检查命令”或“当前推荐从哪个目录运行这些命令”。
6. 如果你在这一步发现明显的命名、导入或示例命令不一致，只修 1 到 2 处最小问题，不要顺手做大规模重构。
7. 今天不要扩展到 pre-commit、CI、完整类型检查、覆盖率门禁或一整套工程模板；只把最小格式化习惯和文档说明立起来。

## 验收标准
1. 项目已明确接入 1 个最小格式化或 lint 工具，优先是 `ruff` 或 `black`。
2. 至少有 1 条命令可以被执行，用来格式化或检查当前最小代码目录。
3. README、练习说明或项目文档中，已补 1 处与该工具相关的运行说明。
4. 如有顺手修正，范围控制在 1 到 2 处小问题，没有扩展成大规模样式清理。
5. 整体工作量控制在 20 到 60 分钟，任务完成后项目更容易继续迭代。
