# 今日任务：实现最小 `POST /task` 输入输出闭环

日期：2026-05-15

## 今日目标
在当前最小 FastAPI 服务里补出一个可调用的 `POST /task` 接口：先只接收一段任务文本，再返回一个结构清晰、便于后续继续演进的结果。今天只要求打通“提交请求 -> 服务处理 -> 返回结构化响应”这条最短路径。

## 为什么做这个
前面几天如果你已经补了服务骨架、`GET /health` 和 `GET /version`，那么下一步最自然的落点就是第一个真正接收输入的接口。`POST /task` 会成为后面接 Pydantic 模型、日志、pytest、tool schema 以及最小 tool-calling flow 的共同入口。就算你前一天没做完，今天也仍然能成立：先补到服务能启动，再把一个最小 POST 接口跑通，依然是当前阶段最划算的一步。

## 任务说明
1. 如果仓库里还没有稳定可启动的 FastAPI 入口，先把服务补到“能启动并接收请求”的程度，再继续做今天的 `POST /task`。
2. 新增一个最小 `POST /task` 路由，只处理一件事：接收用户提交的一段任务文本，并返回结构化结果。不要在今天把它扩展成真实 Agent 编排。
3. 请求体和响应体尽量从现在开始就保持字段命名清晰、结构稳定，方便下一步继续接 Pydantic 请求/响应模型；但如果你今天还没准备用 Pydantic，也可以先把字段设计清楚。
4. 返回结果建议至少体现 3 个信息：服务收到了什么、服务如何标记这次请求、下一步是否可以继续扩展。重点是“结构可读、后续好接”，不是业务复杂。
5. 暂时不要接入外部模型、不要写真实 tool 调用、不要引入复杂异常处理，也不要把日志体系一次性铺开。今天的目标只是最小闭环。
6. 如果 README 里还没写清楚如何启动服务、如何手动调用 `POST /task`，补最少几行说明即可，让今晚的你打开仓库就能直接验证。
7. 如果时间还够，可以顺手补 1 条最小 `pytest` 用例，只验证状态码和核心返回字段；但这不是今天必须完成的部分。

## 验收标准
1. 本地已经有一个可访问的 `POST /task` 接口。
2. 该接口能接收一段任务文本，并返回一个结构明确、字段命名清楚的响应。
3. 整个实现仍然保持“最小可运行”，没有膨胀成模型调用、工具编排或完整异常处理。
4. README 或相关说明里，至少写清楚 1 条启动命令和 1 种手动验证 `POST /task` 的方法。
5. 如果补了测试，`pytest` 至少有 1 条围绕 `POST /task` 的最小用例通过。
