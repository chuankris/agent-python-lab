# 今日任务：给 `/task` 补最小请求日志

日期：2026-06-22

## 今日目标
在你当前的最小 FastAPI 练习里，围绕 `POST /task` 补一层最小可用日志：先保证接口存在，然后让一次请求至少留下 1 到 2 条清晰、可读、不过度泄露输入内容的日志记录。

## 为什么做这个
你前两天已经把重点放在接口骨架、最小测试和 `/task` 的输入输出契约上了。下一步最自然的推进，不是马上把真正的 Agent 调用、tool schema 执行和异常分支一口气塞进去，而是先让这条主链路“可观察”。

这一步很像你以前做 Java/IoT 集成时先把链路打通，再确认日志能不能帮助定位问题。放到 Python/FastAPI 这里，本质没变：先知道请求有没有进来、走到了哪、返回了什么状态，后面你再接 tool schema、最小 tool-calling flow、异常处理和 pytest 覆盖时，排查成本会低很多。

今天只做“最小日志”，不要把任务扩成完整监控系统，也不要顺手引入复杂日志框架。目标是先建立习惯：写接口时，同时考虑最小可观测性。

## 任务说明
1. 继续沿用你已经在做的练习目录，优先使用 `exercises/day-001-project-skeleton/`，不要新开第二套项目。
2. 如果你昨天已经有了 `POST /task`，直接在这条接口上补日志；如果还没有，就先补一个最小占位版 `/task`，能接收最小请求体并返回固定结果即可，然后立刻回到“日志”这个主目标，不要继续扩业务逻辑。
3. 使用 Python 标准库 `logging` 即可。今天的重点不是选库，而是先把日志位置和内容放对。
4. 至少记录两类信息中的一类，最好两类都做：
   - 请求进入 `/task` 时，记录接口名或路径，以及一个足够轻量的关键信息，比如输入文本长度、是否收到请求体、固定状态值。
   - 返回响应前，记录这次请求的处理结果，比如返回状态、固定 task 状态，或是否成功接受。
5. 不要把用户完整输入原文直接打进日志。今天顺手建立一个习惯：优先记录长度、状态、字段是否存在，而不是原始大段文本。
6. 如果你现在把路由、Pydantic 模型和日志配置都写在一个文件里，今天允许做一次很小的整理，例如抽一个简单的日志初始化函数，或把变量命名改清楚；但不要做大重构。
7. 如果时间够，用 `pytest` 补 1 个最小测试，验证 `/task` 正常可调用；如果你已经会用 `caplog`，可以顺手断言关键日志出现过。如果你还不熟，就先只保证接口测试能跑通，日志先靠手动运行观察，不必硬上高级断言。
8. 在练习目录的 `README.md` 里补 2 到 4 行说明：今天增加了什么日志、启动后怎么观察日志、为什么故意不记录完整输入内容。

## 验收标准
1. 本地已经存在可调用的 `POST /task`，即使它现在仍然只是最小占位实现也可以。
2. 调用一次 `/task` 后，终端里能看到至少 1 条与你这次请求明确相关的日志；推荐做到请求开始和请求结束各 1 条。
3. 日志内容没有直接打印整段原始输入，而是采用更克制的信息，例如文本长度、状态值、路径名或固定结果。
4. 至少有 1 个最小验证动作成立：要么是 1 个 `pytest` 用例通过，要么是你能按 README 中的步骤手动触发并观察到日志。
5. 今天的工作量控制在 20 到 60 分钟，没有把任务扩成完整异常处理、真实 tool schema 执行或真正的 Agent 调用。
