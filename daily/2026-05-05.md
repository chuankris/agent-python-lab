# 今日任务：把 `/task` 和最小工具调用串起来

日期：2026-05-05

## 今日目标
让当前项目里的 `/task` 具备一次最小可解释的工具调用流程：接收输入，进入一个明确的工具入口，返回结构化结果。

## 为什么做这个
昨天你已经开始给工具定义 schema。下一步不是马上接真实大模型，而是先把“接口层 -> 工具层 -> 结果层”这条最短链路跑通。这样做的价值很高：你能先练清楚 Python 模块拆分、函数边界、输入输出格式和最小可测试流，而不会过早陷进外部模型、复杂编排或提示词细节。就算你昨天还没完成 schema，今天也可以先补一个最小版本再继续，不会断档。

## 任务说明
1. 选一个最小工具能力，优先继续用 `extract_keywords`，不要换成需要联网、依赖模型或依赖第三方服务的功能。
2. 在项目里放一个清晰的工具入口。形式可以很简单，例如一个函数、一个模块，或者一个单独的 `tools/` 目录；重点是让人一眼看出“工具调用从哪里进、结果从哪里出”。
3. 让 `/task` 或你当前的等价入口接收到文本后，调用这个最小工具入口，并把结果放进响应结构里。
4. 今天的“工具调用”可以是伪调用或本地规则实现，例如先返回固定格式结果，或者做一个非常朴素的关键词提取；重点是流程打通，不是算法效果。
5. 如果你昨天已经写了 schema，今天尽量复用它来约束输入或输出；如果还没写完整，就先补一个最小可用的输入输出定义，再完成串联。
6. 为这条最小 flow 补一条日志，至少能看出一次请求进入了哪个工具，避免后面排查时只能猜。
7. 如果你还有余力，再补 1 个最小测试，验证 `/task` 触发工具调用后返回了预期结构；但如果时间不够，优先保证主流程能清楚跑通。
8. 不要在今天引入真正的 LLM SDK、多工具调度、重试机制或复杂异常体系，这些都留到后面。

## 验收标准
1. `/task` 或等价入口已经能触发一次明确的最小工具调用流程。
2. 响应结果里能看出工具调用后的结构化输出，而不是只回原始文本。
3. 工具入口的位置清晰，后续可以继续扩展成更多工具。
4. 至少有 1 处日志能帮助确认请求经过了工具调用。
5. 整体工作量控制在 20 到 60 分钟内，不追求接入真实模型。
